## 핵심 정체성
- SUPA = 로컬 기반 운영 인프라(Local Operating Infrastructure)
- "얼마나 많이 팔았는가"가 아니라 "이 상권이 구조적으로 얼마나 버틸 수 있는가"를 데이터로 설명하는 인프라 기업
- SaaS가 아님. 상권 운영 OS.

## IR 기획서(docx)와의 차이점
- **IR 기획서**: 투자자 대상, 숫자/수익 중심, P&L 포함, 간결 (AI 경영 자동화 + 공동구매 플랫폼 프레이밍)
- **사업계획서 v5**: 사업 철학/구조 중심, 51페이지 상세, "운영 과정 데이터" 강조, 재무 추정은 의도적으로 배제

---

## 4가지 투자 포인트
1. **Structure**: 무료 진입 + 저수수료 + 대량확장. 데이터 밀도가 투자 포인트
2. **Physical**: 상권 내부 상주형 운영. "앱"이 아니라 "현장 운영 접점"이 핵심
3. **Data**: 매출/결제 같은 결과 데이터 X → 발주, 외주요청, 관리주기 등 "운영 과정 데이터"
4. **Commons**: 영업이익 일부를 상권 기금으로 환원

## 비전/미션
- Vision: 상권의 '생존 확률'을 데이터로 만든다
- Mission: 상권이 '스스로 버티는 운영 구조'를 만들게 한다

---

## 문제 인식
- 상권 문제 = 유입/콘텐츠 부족이 아니라 운영 구조가 점포 단위에 고립
- 잘 되기 시작할수록 더 버거워지는 구조 (인건비, 외주 의존도, 복잡도 증가)
- 기존 플레이어(POS, PG, 배달앱)는 결과 데이터만 보유. 운영 과정 데이터는 공백

## 시장 공백
- 배달앱/POS/PG = "얼마가 결제되었는가" (결과)
- SUPA = "왜 운영이 불안정해지는가" (과정)
- 이 과정 데이터는 상권 내부에 상주하지 않으면 축적 불가 → 진입장벽

---

## 서비스 모듈 (3개)

### SUPA CLUB (Core) — 비용 구조 안정화 엔진
- 공동구매 기반, 반복 지출을 상권 단위로 재편
- 상위 5~10개 반복 소모품 SKU 중심
- "쓰면 무조건 이득" = 진입 엔진
- 축적 데이터: SKU별 구매주기, 점포유형별 소비패턴, 급구 발생빈도

### SUPA SQUAD (Orchestration) — 운영 판단 구조화 레이어
- 외주/전문 영역 요청을 '요청→조건→선정→실행→검수→기록'으로 구조화
- 직접 수행 X, 연계/관리/기록 중심 → 점포 수 증가해도 인력 선형 증가 안 함
- 축적 데이터: 운영요청 패턴, 긴급요청 비중, 외주 의존도, 벤더 성과

### SUPA CLEAN (Expansion) — 리스크 관리 확장 모듈
- 위생/시설 관리를 단건 대응 → 누적 관리 체계로 전환
- 관리 항목/주기/검수 기준 표준화
- 축적 데이터: 관리주기 편차, 리스크 발생 패턴, 설비 취약 유형

## 7단계 성장 경로

| 단계 | 시기 | 내용 |
|---|---|---|
| 1. 비용 혁신 | Y0~1 | 공동구매 + 외주구조화 + 관리표준화. 무과금 |
| 2. 운영 루틴 형성 | Y1~2 | 급구 감소, 외주 구조화, 관리 사전예방화 |
| 3. 데이터 구축 | Y2~3 | 운영 과정 데이터 축적. "상권이 얼마나 안정적인가" 설명 가능 |
| 4. 상권 단위 데이터 | Y3~4 | 점포→상권 단위 분석. 결제/금융 구조 검토 |
| 5. 상권 기금 조성 | Y3~4 | 영업이익 5% 환원. 공동 인프라 재투자 |
| 6. 상권 기획 | Y4~5 | 지원 사업 이후를 설계하는 주체 |
| 7. AI/데이터 비즈니스 | Y5+ | 예측, 사전감지, 의사결정 자동화 보조 |

## 수익 구조 (5단계)
1. **공동구매/유통 리베이트** (Phase 1) — 무과금, 거래량 비례 자연 발생
2. **벤더/전문가 연계 수수료** — 절감 구조 안에서만 소규모 수수료
3. **결제 수수료 절감** (Phase 2) — 1,000~3,000개 점포 이후 PG/POS 재협상
4. **구독료 = Data Access Fee** — 운영 인프라로 인식된 후 선택적 도입
5. **AI 기반 고도화** (Phase 3) — 예측/감지/자동화, 성과 연동 과금

## KPI 구조 (3축)
1. **침투도**: 가입 점포 수, 활성 점포 비율, 비용 절감 체감 지표
2. **활성도**: 반복 이용률, 월간 운영 트랜잭션 수, 벤더 품질 지표
3. **데이터 밀도**: 패턴/기준선 도출 가능 수준

## BEP 정의
- 기존 "점포 수 x 구독료" 폐기
- 새 정의: 운영 트랜잭션 x 처리 효율 x 데이터 밀도

## 타깃 시장
- TAM 기준 설정 X → "같은 문제를 반복적으로 겪는 구조"가 기준
- 1차: 행궁동 (300~500개소) → 수원 (1,000개소)
- 2차: 유사 구조 관광/생활 혼합 상권 (구도심, 역사문화 상권)
- 확장 기준: 구조가 반복되지 않으면 확장하지 않음

## MVP 구성
- ENTRY: 점포 촬영 + 1페이지 진단 피드백 (계약 전 체감)
- CLUB: 상위 5~10개 SKU 공동구매
- CLEAN: 에어컨/덕트 등 핵심 1~2개 관리
- SQUAD: 벤더 요청 접수/구조화/검수/기록
- 월 1회 운영 데이터 리포트

## 기획서 활용 포인트
- IR 기획서(docx)가 "투자자에게 매력적인 숫자"라면, 이 문서는 "사업의 철학과 구조적 깊이"
- "운영 과정 데이터"라는 독자적 포지셔닝 = 경쟁사와의 근본적 차별점
- 7단계 성장 경로 = 장기 비전의 구체적 로드맵
- SPC 사례 (20%→8% 수수료 절감) = 비용 혁신의 현실적 근거
- 상권 기금 = 단순 CSR이 아닌 데이터 품질/사업 안정성 강화 장치
