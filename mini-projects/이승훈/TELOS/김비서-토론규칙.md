# 김비서 — 토론 구조 (GEM 어벤져스)

> 이 파일은 GEM 토론이 필요할 때만 읽는다. 매 세션 로드 대상 아님.

---

## 실행 방식: 독립 에이전트 기반

GEM은 페르소나(역할극)가 아니라 **독립된 에이전트**로 실행한다.
각 GEM은 자기만의 세션에서 작업하고, 결과만 김비서에게 보고한다.

**현재 에이전트화 완료된 GEM:**
- 영희 (데이터) → `~/.claude/agents/gem-younghee.md`
- 미애 (브랜드) → `~/.claude/agents/gem-miae.md`

**아직 페르소나인 GEM:** 철수, 주희, 미영, 진희 (필요 시 에이전트로 전환)

**에이전트 소집 방법:** Agent 도구로 해당 GEM 에이전트를 서브 에이전트로 호출한다.
- 서브 에이전트는 자기만의 컨텍스트를 사용 → 메인 세션 용량 보호
- 서브 에이전트에게 안건 + 체크리스트 결과를 전달
- 서브 에이전트가 분석 결과를 파일로 저장 → 김비서가 파일만 읽음

## 토론 흐름 (3라운드, 에이전트 기반)

```
승훈님: 작업 요청
    |
김비서: 안건 파악 + 체크리스트 채우기
    |
[1라운드] 각 GEM 에이전트를 서브 에이전트로 소집 (병렬 실행)
          → 각자 독립 세션에서 분석
          → 결과를 research/ 에 파일로 저장
    |
[2라운드] 김비서가 GEM 결과 파일들을 읽고 크리틱 포인트 정리
          → 크리틱 포인트를 다시 GEM 에이전트에게 전달
          → GEM이 크리틱 반영하여 보완안 파일 저장
    |
[3라운드] 김비서가 보완안들을 종합하여 최종안 작성
    |
김비서: 승훈님에게 보고
  - 최종 권고안 (추천)
  - GEM 간 이견이 있었던 부분과 결론
  - 선택지 2~3개
    |
승훈님: 선택 또는 추가 수정
```

## 루프 강화 (기본 3회)

GEM 토론은 기본적으로 **3회 루프**를 돌린다:
- 1회: 초안 생성
- 2회: 초안을 다시 GEM에게 넣어서 크리틱 → 개선
- 3회: 2회 결과를 다시 넣어서 최종 다듬기
- 파일 이름은 항상 동일하게 유지 (덮어쓰기) → 진짜 재귀적 개선
- 각 루프의 변경 사항을 별도 로그로 기록
- 승훈님이 "루프 N회"라고 하면 횟수 조정 가능

## 안건별 GEM 소집 (자동)

| 안건 | 소집 GEM |
|---|---|
| IR 업데이트 | 영희(데이터) + 철수(밸류) + 미애(브랜드) |
| 사업계획서 업데이트 | 영희(데이터) + 철수(밸류) + 미영(조직) |
| 브랜드/마케팅 | 미애(브랜드) + 영희(데이터) + 진희(AI) |
| 계약/법률 | 주희(법률) + 철수(밸류) |
| 조직/프로세스 | 미영(조직) + 진희(AI) |
| 김비서 시스템 | 미영(조직) + 진희(AI) |
| 소명서, 회의록 등 단순 문서 | 토론 없이 바로 실행 |

## 로하이사회 모드 (전원 소집)

중대한 의사결정일 때 발동. "이사회 소집해줘"로 수동 발동 가능.
5단계 프로세스: 안건 상정 -> GEM 브리핑 -> 크리틱 -> 균형점 탐색 -> 최종 권고안.
상세 규칙: `TELOS/GEM-어벤져스.md` 참조.
