## 0. 현재 기술 자산 진단

코드를 직접 확인한 결과, 현재 공존공간의 기술 스택은 다음과 같다.

| 구분 | 내용 | 성숙도 |
|------|------|--------|
| **오케스트레이터** | monthly-report.js -- 업체명+월 입력하면 노션 페이지 자동 생성 | 작동 중 |
| **모듈 구조** | businesses/{업체명}/config.js, collectors.js, sheets.js | 확장 가능 설계 |
| **공통 라이브러리** | lib/notion.js, blocks.js, weeks.js | 안정화 |
| **데이터 수집** | okpos-sales.js(POS), water-bill.js(수도요금), scan-mail.js(메일 파싱) | 작동 중 |
| **웹서버** | server.js (Express, 포트 3001) -- PDF 업로드 + 사업 DB CRUD | 기초 단계 |
| **데이터베이스** | gongzon.db (SQLite, better-sqlite3) + 노션 DB 5개 | 하이브리드 |
| **자동화 도구** | Puppeteer(웹 스크래핑), imap+mailparser(메일), multer(파일) | 도구 수준 |

비유하자면: **주방은 갖춰져 있고, 한 가지 요리(월정산)는 잘 만들고 있다.** 문제는 "이 주방에서 10가지 요리를 동시에 만들 수 있는가?"이다.

## 1. 기술 로드맵: Phase별 무엇이 필요한가

### Phase 1 (0~3개월): 내부 완성 -- "지금 주방을 더 효율적으로"

현재 월정산 자동화를 완성하고, 임차인에게 보여줄 수 있는 첫 화면을 만든다.

| 기능 | 기술 요구사항 | 현재 스택 활용도 | 난이도 |
|------|-------------|-----------------|--------|
| 월정산 자동화 완성 | 현재 코드 안정화 + 에러 핸들링 | 100% | 낮음 |
| 임차인 열람용 웹 대시보드 | Express + 정적 HTML/CSS 또는 경량 프레임워크 | server.js 확장 | 중간 |
| 공과금 자동 수집 확대 | scan-mail.js 확장 (전기, 가스 등) | 80% | 낮음 |
| PDF 보고서 자동 생성 | puppeteer로 노션 페이지를 PDF 변환 | 도구 있음 | 낮음 |

**핵심 판단**: 이 단계는 현재 스택(Node.js + SQLite + 노션 + Express)으로 충분하다. 새 기술을 도입할 필요 없다.

### Phase 2 (3~6개월): 임차인 도구 -- "손님에게 메뉴판을 주자"

임차인이 직접 사용하는 기능을 하나씩 추가한다.

| 기능 | 기술 요구사항 | 자체 개발 vs 연동 | 난이도 |
|------|-------------|------------------|--------|
| 급여 자동 계산 | 주휴수당/야간수당 로직 + 4대보험 요율표 | **연동 권장** | 높음 |
| 근태 관리 (출퇴근) | GPS/QR 인증 모바일 앱 | **연동 권장** | 높음 |
| 간편 장부 | 매출/비용 입력 + 손익 자동 계산 | 자체 개발 가능 | 중간 |
| 매출 현황 대시보드 | 차트 라이브러리 (Chart.js 등) + API | 자체 개발 가능 | 중간 |

**핵심 판단**: 여기서 갈림길이 생긴다. 급여/근태/세무는 **법률이 바뀔 때마다 업데이트해야 하는 영역**이다.

### Phase 3 (6~12개월): AI 활용 + 플랫폼화 -- "요리사에게 AI 어시스턴트를"

| 기능 | 기술 요구사항 | 데이터 준비도 | 난이도 |
|------|-------------|-------------|--------|
| 매출 예측 | 시계열 분석 (Prophet 등) | POS 데이터 축적 필요 (최소 12개월) | 높음 |
| 공실 예측 | 매출 트렌드 + 임대 계약 데이터 | 데이터 부족 | 매우 높음 |
| 자동 발주 제안 | 매출-재고 상관 분석 | 재고 데이터 없음 | 높음 |
| 매출보전보험 데이터 | 매출 변동성 통계 파이프라인 | POS 데이터 있음 | 중간 |

### Phase 4 (12개월+): 양면 플랫폼 -- "다른 건물에도 주방을 열자"

| 기능 | 기술 요구사항 | 현재 준비도 |
|------|-------------|------------|
| 멀티 테넌트 아키텍처 | DB 분리 or 테넌트 격리 설계 | SQLite로는 한계 |
| 사용자 인증/권한 | OAuth 또는 자체 인증 시스템 | 현재 없음 |
| API 게이트웨이 | 외부 서비스 연동 허브 | 현재 없음 |
| 결제/과금 시스템 | 구독료 수납, PG 연동 | 현재 없음 |

**핵심 판단**: Phase 4는 현재 스택의 "대수술"이 필요하다. SQLite에서 PostgreSQL/MySQL로 전환, 인증 시스템 구축, 클라우드 배포 등. 이것은 Phase 1~3을 통해 검증된 후에 판단해도 늦지 않다.

## 2. 자체 개발 vs 연동: "만들 것인가, 빌릴 것인가"

이 질문에 대한 판단 기준은 세 가지다:

1. **법률 변경에 민감한가?** -- 민감하면 연동 (전문 업체가 업데이트)
2. **우리만의 차별점인가?** -- 차별점이면 자체 개발
3. **비개발자 팀이 유지보수 가능한가?** -- 불가능하면 연동

| 영역 | 판정 | 근거 |
|------|------|------|
| **월정산 자동화** | 자체 개발 (유지) | 핵심 차별점. 이미 작동 중. 건물주-임차인 관계를 이해하는 유일한 시스템 |
| **급여 계산** | 연동 권장 | 주휴수당 계산식만 해도 예외 케이스가 수십 가지. 최저임금/4대보험 요율이 매년 바뀜. 보스몬(무료) 또는 얼마에요(4인까지 무료)를 안내하는 것이 합리적 |
| **근태 관리** | 연동 권장 | GPS/QR 출퇴근은 모바일 앱 개발이 필수. 보스몬+크루몬 조합이 무료이고 완성도 높음 |
| **세무/장부** | 연동 권장 | 세법은 매년 바뀌고, 홈택스 API 연동은 인증서 처리가 복잡. 삼쩜삼/SSEM/얼마에요가 이미 해결 |
| **임차인 대시보드** | 자체 개발 | 이것이 플랫폼의 "얼굴". 건물주가 임차인에게 제공하는 가치 그 자체 |
| **매출 데이터 수집** | 자체 개발 (유지) | okpos-sales.js가 이미 작동. POS별 수집기를 모듈로 확장 가능 |
| **매출 예측 AI** | 자체 개발 (단, Phase 3) | 데이터가 쌓이면 공존공간만의 인사이트가 됨. 외부 서비스에 맡기면 데이터를 남에게 주는 꼬리 |

비유하자면: **급여/세무/근태는 "전기"다. 직접 발전소를 지을 게 아니라 한전에서 끌어오면 된다.** 우리가 만들어야 하는 건 **"건물 관리라는 요리"** 자체다.

### 연동 전략: "있는 것을 잘 연결하자"

```
[공존공간 대시보드]
     |
     |-- 매출 데이터: okpos-sales.js (자체, 이미 있음)
     |-- 공과금: scan-mail.js (자체, 이미 있음)
     |-- 정산 보고서: monthly-report.js (자체, 이미 있음)
     |
     |-- 급여/근태: "보스몬 쓰세요" (안내 + 링크)
     |-- 세무/장부: "얼마에요 쓰세요" (안내 + 링크)
     |-- 매출 분석: "캐시노트 쓰세요" (안내 + 링크)
```

당장은 API 연동이 아니라 **"이 도구를 쓰세요"라고 안내하는 것**만으로 충분하다. 보스몬도, 얼마에요도 API를 공개하지 않고 있기 때문에, 기술적 연동보다 "큐레이션"이 현실적이다.

## 3. AI 활용 기회: "이 중에 AI가 대신할 수 있는 건 뭔가?"

### 지금 당장 가능한 것 (데이터와 도구가 있음)

| AI 활용 | 방법 | 데이터 | 효과 |
|---------|------|--------|------|
| **공과금 메일 자동 파싱** | scan-mail.js + LLM API (Claude/GPT) | 이메일 수신함 | 수동 입력 제거. 현재 정규식 파싱을 LLM으로 바꾸면 형식 변경에 강해짐 |
| **월정산 보고서 자동 요약** | 정산 데이터를 LLM에 넣어 텍스트 요약 | 노션 DB 데이터 | "이번 달 매출이 전월 대비 8% 감소, 주된 원인은 2주차 비수기" 같은 인사이트 |
| **임차인 문의 자동 응답** | FAQ 기반 챗봇 (RAG 구조) | 임대 계약서, 규정, 과거 문의 | 반복 질문 처리 시간 절감 |

### 6개월 후 가능한 것 (데이터 축적 필요)

| AI 활용 | 필요 데이터 | 최소 데이터량 | 기대 효과 |
|---------|------------|-------------|----------|
| **매출 예측** | 일별 POS 매출 + 요일/날씨/이벤트 | 12개월 이상 | 비수기 대비, 인력 배치 최적화 |
| **공과금 이상 탐지** | 월별 공과금 추이 | 6개월 이상 | 수도/전기 누수 조기 발견 |
| **임차인 이탈 예측** | 매출 추이 + 계약 잔여기간 | 12개월 이상 | 공실 전에 선제 대응 |

### 12개월 후 가능한 것 (플랫폼 데이터 필요)

| AI 활용 | 필요 조건 | 기대 효과 |
|---------|----------|----------|
| **매출보전보험 리스크 모델** | 다수 매장 매출 데이터 | 보험 상품 설계의 기초 통계 |
| **상권 분석 기반 임대료 적정가** | 지역별 매출/임대료 비교 데이터 | 건물주의 임대 의사결정 지원 |
| **자동 발주 제안** | 매출-재고 상관 데이터 | 식재료 폐기 감소, 원가 절감 |

### AI 활용의 제약 원칙

- **인간적 상호작용을 대체하지 않는다**: 임차인과의 관계, 문제 상담, 계약 협상은 사람이 한다
- **블랙박스 최소화**: "왜 이런 예측이 나왔는지" 설명할 수 있어야 한다. 설명 불가능한 AI 모델은 사용하지 않는다
- **AI는 결정하지 않는다**: 예측을 보여주고, 사람이 판단한다. "AI가 공실 예측했으니 임대료 낮추세요"가 아니라 "이 데이터를 보시면 판단에 도움이 됩니다"

## 4. 기술 리스크: "비개발자 + AI 비서 팀으로 가능한가?"

### 솔직한 현재 역량 평가

| 역량 | 현재 수준 | 근거 |
|------|----------|------|
| Node.js 스크립팅 | 중급 | monthly-report.js, okpos-sales.js 등 작동하는 코드가 있음 |
| 웹 서버 | 초급 | server.js가 Express 기본 구조만 사용 |
| 데이터베이스 | 초급 | SQLite 단일 파일. 마이그레이션/백업 체계 없음 |
| 프론트엔드 | 없음 | public/ 폴더에 정적 파일만 |
| 모바일 앱 | 없음 | 앱 개발 경험 없음 |
| AI/ML | 없음 | LLM API 호출은 가능하지만, 모델 학습/튜닝은 미경험 |
| DevOps/배포 | 없음 | 로컬 실행만. 클라우드 배포 경험 없음 |

### 리스크 매트릭스

| 리스크 | 발생 가능성 | 영향도 | 대응 |
|--------|-----------|--------|------|
| **Phase 1~2 지연** | 중간 | 높음 | AI 비서(Claude Code)와 함께 작업하면 관리 가능. 현재 패턴 유지 |
| **급여 계산 오류 → 법적 문제** | 높음 | 매우 높음 | **자체 개발하지 않는다.** 보스몬/얼마에요 연동으로 회피 |
| **데이터 유실** | 중간 | 매우 높음 | SQLite 백업 자동화 + 노션이 2차 저장소 역할. git으로 코드 보호 |
| **Phase 4 기술 부채** | 높음 | 중간 | Phase 3 끝나고 판단. 그때 외부 개발자 영입 or 기술 파트너 필요 |
| **보안 사고 (데이터 유출)** | 낮음 | 매우 높음 | 현재 로컬 실행이라 외부 노출 없음. 웹 공개 시 인증 필수 |
| **AI 비서 의존도 과다** | 중간 | 중간 | 코드 구조를 단순하게 유지하고, 주석과 문서를 충실히 작성 |

### 핵심 판단: 가능한가?

**Phase 1~2는 가능하다.** 현재 팀 구성(대표 + AI 비서)으로 월정산 완성 + 임차인 대시보드까지는 도달할 수 있다. 이유:
- 이미 작동하는 코드 자산이 있다
- Node.js + Express + SQLite는 비개발자가 AI와 함께 다루기에 적합한 스택이다
- 급여/세무/근태 같은 위험 영역은 외부 서비스로 회피한다

**Phase 3~4는 팀 확장이 필요하다.** AI/ML 모델 구축, 멀티 테넌트 아키텍처, 클라우드 배포, 보안 설계는 전문성이 필요하다. 그때가 되면:
- 프리랜서 개발자 1명 (백엔드/인프라)
- 또는 기술 공동 창업자
- 또는 노코드/로코드 플랫폼으로 우회 (Bubble, Supabase 등)

## 5. MVP 제안: "가장 빠르게 임팩트를 내는 첫 단계"

### 제안: "임차인이 볼 수 있는 월정산 웹페이지" (2~4주)

현재 monthly-report.js가 만드는 노션 페이지를, **임차인이 웹 브라우저에서 열어볼 수 있는 페이지로 바꾸는 것**이다.

```
지금:  node monthly-report.js gongzon 2026-03
       → 노션에 페이지 생성
       → 건물주가 캡처해서 카톡으로 전달

MVP:   같은 명령어 실행
       → 노션 페이지 + 웹 페이지 동시 생성
       → 임차인에게 URL 하나 보내기
       → 임차인이 브라우저에서 이번 달 정산 확인
```

### 왜 이것이 "첫 단계"인가

1. **기존 코드 90% 재사용**: monthly-report.js의 데이터 수집 로직은 그대로. 출력만 노션 API 대신 HTML로 추가
2. **server.js가 이미 있다**: Express 서버가 포트 3001에서 돌고 있다. 라우트 하나 추가하면 된다
3. **임차인의 반응을 즉시 확인**: "이게 편하다 / 필요없다 / 이것도 보고 싶다"를 알 수 있다
4. **플랫폼의 씨앗**: 이 URL이 나중에 "임차인 대시보드"의 첫 페이지가 된다

### 기술 구현 경로

```
1. server.js에 /report/:business/:month 라우트 추가
2. monthly-report.js의 데이터 수집 함수를 API로 노출
3. 간단한 HTML 템플릿으로 정산 데이터 렌더링 (표 + 합계)
4. 임차인에게 링크 전달: http://localhost:3001/report/gongzon/2026-03
   (나중에 도메인 연결)
```

필요한 새 기술: **없음**. Express + HTML + 기존 데이터 수집 코드.

## 6. 강점 / 약점 / 개선 제안

### 강점 3가지

1. **모듈화된 코드 구조**: businesses/{업체명}/ 구조는 새 건물/업체 추가가 "폴더 복사 + 설정 변경"으로 가능하게 설계되어 있다. 이것은 플랫폼 확장의 기술적 토대다.

2. **실데이터 기반 자동화가 이미 작동한다**: POS 매출 수집, 메일 파싱, 수도요금 자동 수집이 실제로 돌아간다. 이론이 아니라 실전이다. 경쟁사 분석에서 본 보스몬/얼마에요도 데이터 수집 자동화가 핵심인데, 공존공간은 "건물주 관점의 데이터 수집"이라는 남들이 안 하는 영역을 이미 해결하고 있다.

3. **하이브리드 데이터 구조(SQLite + 노션)가 실용적이다**: SQLite로 구조화된 데이터를 관리하고, 노션으로 사람이 읽을 수 있는 보고서를 만드는 조합은 비개발자 팀에 매우 적합하다.

### 약점 3가지

1. **웹 배포/보안 인프라가 없다**: server.js가 localhost에서만 돌고, 인증/권한 관리가 없다. 임차인에게 URL을 공유하려면 최소한의 배포와 접근 제어가 필요하다.

2. **에러 핸들링과 모니터링이 부족하다**: POS 사이트가 바뀌거나 메일 형식이 바뀌면 스크립트가 조용히 실패할 수 있다. "실패했다"는 것을 누가 알아차리는가?

3. **프론트엔드 역량이 사실상 제로다**: 임차인에게 보여줄 화면을 만드는 능력이 없다. 대시보드, 차트, 모바일 대응 등은 현재 팀으로는 AI 비서에 전적으로 의존해야 한다.

### 개선 제안 3가지

1. **에러 알림 시스템 구축 (1일 작업)**: 각 수집 스크립트에 try-catch를 추가하고, 실패 시 카카오톡 또는 슬랙으로 알림을 보내는 함수 하나를 만든다. "조용한 실패"를 막는 것이 자동화의 신뢰를 지키는 핵심이다.

2. **SQLite 자동 백업 (반나절 작업)**: gongzon.db를 매일 새벽에 날짜별로 복사하는 배치 파일(bat)을 만든다. 지금 .db-wal, .db-shm 파일이 보이는데, WAL 모드로 운영 중이라면 백업 시 체크포인트가 필요하다.

3. **"임차인 열람 MVP"를 AI 비서와 2주 안에 완성하라 (이번 Phase 1의 다음 마일스톤)**: 완벽한 대시보드가 아니라, "이번 달 정산 숫자가 표로 보이는 웹페이지 한 장"이면 충분하다. 임차인의 반응을 보고 나서 다음을 결정한다.

## 7. 최종 의견: 기술 관점 총평

> "기술로 더 잘할 수 있는가?"

**예. 단, "모든 것을 기술로 만드는 것"이 아니라 "핵심만 만들고 나머지는 연결하는 것"이 정답이다.**

공존공간의 기술적 해자(moat)는 "건물주-임차인 사이의 정산 데이터를 자동으로 만드는 것"이다. 급여 계산은 보스몬이 잘한다. 장부는 얼마에요가 잘한다. 매출 분석은 캐시노트가 잘한다. 하지만 **"건물주가 임차인의 운영 현황을 한눈에 보고, 임차인이 이번 달 정산을 웹으로 확인하는 것"은 아무도 안 하고 있다.**

기술 투자의 우선순위:
1. **지금**: 월정산 완성 + 임차인 웹 열람 (기존 스택으로 가능)
2. **3개월 후**: LLM 연동으로 데이터 파싱 고도화 + 보고서 자동 요약
3. **6개월 후**: 매출 예측 PoC (데이터 12개월 축적 시점)
4. **판단 유보**: 양면 플랫폼 아키텍처는 Phase 2 완료 후 시장 반응을 보고 결정

슈퍼앱이라는 단어에 홀리지 말자. **Square도 카드리더기 하나에서 시작했고, Toast도 레스토랑 POS 하나에서 시작했다.** 공존공간은 "월정산 자동화"라는 바늘귀를 이미 통과했다. 다음은 그 실에 무엇을 꿰느냐의 문제다.
