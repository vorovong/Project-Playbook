# 로하이사회 2라운드 -- 미영(프로세스 혁신) 재평가

> 안건: 슈파 v5 리프레이밍 (AI 전략 참모) + MVP v0.1 스펙
> 일자: 2026-04-07
> 렌즈: 실행할 수 있는가? 프로세스는?
> 전제: 4/6 이사회 이후 슈파가 "양면 슈퍼앱"에서 "AI 전략 참모"로 축소됨

---

## 0. 결론

**조건부 실행 가능 (수정 필요).**

축소는 옳다. 4/6 때 내가 "슈퍼앱은 3+AI로 불가"라고 했던 그 우려는 v5에서 상당 부분 해소됐다. 슈파는 더 이상 플랫폼이 아니고, **이승훈 1명이 종이 한 장 들고 사장님 만나는 컨설팅 실험**이다. 이 정도면 3+AI 자원 안에 들어온다.

**그러나** v0.1 스펙은 여전히 이승훈 단일 장애점을 그대로 안고 있고, 한 사이클의 시간 견적이 빠져 있다. 견적을 박지 않으면 "어느 달에 미식가 못 봄"이 반복되며 피드백 루프가 끊긴다. 이걸 잡지 않으면 v5도 낙서가 된다.

**내일부터 실행하라고 하면 첫 단계:**
미식가의주방 1회차 리포트를 만드는 데 **이승훈이 실제로 몇 시간 쓰는지** 타임박스 측정. 추정 말고 실측. 이거 없이는 월 1회 사이클이 지속 가능한지 누구도 모른다.

## 2. 이승훈 부하 분석

### 2-1. 슈파 v0.1이 추가하는 일

| 작업 | 빈도 | 1회 추정 시간 [추정] |
|---|---|---|
| 외부 데이터 수집 (KT, 신한카드) | 월 1회 | 2~3시간 |
| 내부 데이터 추출 (OKPOS, 네이버 예약) | 월 1회 | 1~2시간 |
| 데이터 정합/매칭 (날짜+시간+인원 조인) | 월 1회 | 1~2시간 |
| AI 프롬프트 설계·튜닝 (초기 3회) | 월 1회 | 3~5시간 |
| 리포트 자가 검증 ("내가 봐도 이해되나") | 월 1회 | 1시간 |
| 사장님 미팅 (이동·진행·기록) | 월 1회 | 2시간 |
| 피드백 정리 + 다음 리포트 수정 사항 메모 | 월 1회 | 1시간 |
| **합계 (1사이클)** | **월 1회** | **11~16시간** |

→ **월 1.5~2일치 작업**. 이것이 6개월 동안 지속.

### 2-2. 다른 업무와의 충돌

4/6에서 내가 권고한 시간 배분: 개발 40% / 운영 30% / 영업 30%.
슈파 v0.1은 이 중 **개발 40% 안에 들어가야 한다.** 주 12시간 × 4주 = 월 48시간이 개발 예산이다.

| 개발 예산 (월 48시간) | 사용처 |
|---|---|
| Phase 1 잔여 (월정산 안정화) | 8~10시간 |
| Phase 2 (데이터 축적·리포팅) | 15~20시간 |
| **슈파 v0.1 1사이클** | **11~16시간** |
| 여유 (장애 대응, 학습) | 5~10시간 |

→ **빡빡하지만 산술적으로는 들어간다.** 단, 한 가지 전제 하에서만: **공존공간 운영 위기·정부사업 마감 같은 외부 충격이 없을 때.** 1년 중 그런 달이 절반쯤 된다는 게 4/6의 진단이었다.

**판정:** 충돌 가능. 단, 슈파 v0.1을 "지키는 1순위"가 아니라 "예산 남으면 굴리는 학습 실험"으로 정의하면 감내 가능.

## 3. 월 1회 사이클의 현실성

### 3-1. 1사이클 단계별 시간 견적 (위 표 합산)

**11~16시간 / 사이클**, 사이클 내 분포:

```
[1주차] 데이터 수집 (3~5h) ─ 가장 무거움
[2주차] 매칭 + 프롬프트 튜닝 (4~7h) ─ AI 검증의 핵심
[3주차] 자가 검증 + 미팅 + 피드백 (4h)
[4주차] 다음 사이클 준비 + 휴지
```

### 3-2. 월 1회는 충분한가?

**부족하다.** 이유:
1. 사장님이 액션을 실행하는 데 1~2주 걸린다고 가정하면, **다음 미팅까지 2~3주가 비어버린다.** 그동안 사장님은 슈파를 잊는다.
2. L1→L2→L3의 학습 루프가 너무 길다. 6개월 = 6사이클. 프롬프트 언어를 6번밖에 못 고친다. 부족하다.
3. AI 제안 언어 검증은 빈도 ≫ 깊이다. **격주 30분 미팅**이 월 1회 2시간 미팅보다 학습이 빠르다.

**권고:** 1사이클 = 격주(2주). 단, 첫 1사이클은 풀버전(2시간), 2주 뒤는 가벼운 follow-up(30분, 카톡 가능). 그러면 월 2회 접점 + 부하는 ~12시간/월로 거의 동일.

## 4. 프로세스 병목 진단

### 4-1. 4개 데이터 소스 수집의 단계별 [추정]

| 소스 | 작업 | 추정 시간 | 자동화 난이도 |
|---|---|---|---|
| KT 빅데이터 | 웹 로그인 → 지역·기간 설정 → CSV 다운로드 | 30~60분 | 중 (스크래핑 가능) |
| 신한카드 빅데이터 | 동일 패턴 | 30~60분 | 중 |
| OKPOS | 기존 `okpos-sales.js` 재활용 | 5분 (이미 자동) | 완료 |
| 네이버 예약 | API 미확인 → 수동 추출 가능성 | 60~90분 | **상 (가장 큼)** |

### 4-2. 가장 큰 병목

**병목 #1: 네이버 예약 데이터 추출.**
공식 API 유무가 v0.1 스펙의 오픈 이슈로 남아있다. 이게 결정 안 되면 매월 1~2시간이 "엑셀 복붙"으로 증발한다. 최악의 경우 사이클 자체가 멈춘다.

**병목 #2: AI 프롬프트 튜닝.**
초기 3사이클은 매번 프롬프트를 손봐야 한다. 이건 시간이 아니라 **인지 부하**다. 데이터 수집은 멍 때리며 할 수 있지만, 프롬프트 튜닝은 집중력 풀로드. 같은 날 둘 다 하면 둘 다 망함.

**병목 #3: 자가 검증의 솔직함.**
"내가 봐도 이해되는가"는 만든 사람이 가장 못하는 질문이다. 박승현 또는 엄수빈을 **30분짜리 알파 독자**로 끌어들여야 한다. 사람 1명만 더 읽으면 L1 실패 위험이 절반으로 줄어든다.

### 4-3. 자동화 우선순위 (v0.1은 수동 OK, 어디부터?)

```
1순위 (즉시): 네이버 예약 추출 방식 결정. API든 스크래핑이든 1주일 안에 박아라.
              → 안 박으면 매월 가장 무거운 작업이 됨.
2순위 (3사이클 후): KT/신한카드 다운로드 자동화.
                   → 패턴이 안정된 후 1회만 짜면 영구 회수.
3순위 (L3 도달 후): 매칭 로직 자동화.
                   → 그전엔 수동이 더 빠름. 변수가 많음.
4순위 (보류): AI 프롬프트 자동화.
              → 자동화하면 안 됨. 이게 자동화되는 순간 검증의 본질이 사라진다.
```

**원칙: v0.1에서 자동화할 것은 데이터 수집뿐. AI 레이어와 미팅은 절대 자동화 금지.** 슈파의 검증 대상이 거기에 있다.

## 5. L1→L2→L3 측정 프로세스 설계

**현재 v0.1 스펙의 문제:** "표정·질문·반응 관찰"로만 적혀있다. 이건 측정이 아니라 인상이다. 다음 미팅 때까지 사라진다. 기록 없으면 피드백 루프 실패.

### 5-1. 미팅 1회당 산출물 (필수)

**A4 1장 미팅 노트** — 사장님 만난 직후 30분 안에 채운다. 며칠 지나면 절반 잊는다.

```
[미팅 메타]
- 일시 / 장소 / 동석자
- 리포트 버전 (v0.1.3 등)

[L1 — 이해]
□ 사장님이 끝까지 들었다 (Y/N)
□ 도중에 질문이 있었다 → 질문 내용 그대로 인용:
□ "이해됐어"에 준하는 발화 → 인용:
판정: L1 도달 / 부분 / 미도달

[L2 — 동의]
□ 액션 1 ("___") → 반응:
□ 액션 2 ("___") → 반응:
□ 액션 3 ("___") → 반응:
□ "해볼게" 발화 여부 → 어떤 액션에:
판정: L2 도달 / 부분 / 미도달

[L3 — 실행] (다음 미팅에서 채움)
□ 실제로 실행한 액션:
□ 실행 시점:
□ 실행 후 사장님 소감:
판정: L3 도달 / 미도달

[프롬프트 개선 메모 — 다음 사이클 반영]
- 이해 막힌 단어/문장:
- 빼야 할 부분:
- 더해야 할 부분:
```

### 5-2. 기록 방식

- **녹음 (필수)**: 사장님 동의 받고 폰 녹음. 사람 기억은 못 믿는다.
- **노트 (필수)**: 위 템플릿. 노션 또는 Project_Playbook 안에 `meetings/` 폴더.
- **요약 (선택)**: 녹음을 김비서에게 던져 자동 요약 → 노트 보조.

### 5-3. 합격선 자동 추적

```
미팅 노트 6개 누적 시점에 다음을 확인:
- L1 도달 횟수 / 6
- L2 도달 횟수 / 6
- L3 도달 횟수 / 6 ← 이게 1 이상이면 MVP 합격
```

**김비서에게 월말 자동 집계 요청.** 사람이 까먹는다.

## 6. 단순화 권고 3가지

### 권고 1: 첫 사이클은 데이터 1쌍만 써라

KT + 신한카드 + OKPOS + 네이버 예약 = 4개. 첫 사이클부터 4개 다 합치려 하면 데이터 수집·매칭에 사이클의 80%가 들어간다. **첫 사이클은 OKPOS + KT 2개만.** 그것만으로도 "이 시간대 행궁동에 OO 인구가 있는데 우리 매출은 OO다 → 갭이 OO다"가 나온다. 충분한 깊이 2 리포트가 된다. 검증 대상은 데이터 양이 아니라 **AI 제안 언어**다.

→ 2사이클부터 신한카드 추가, 3사이클부터 네이버 예약 추가. 점진적으로.

### 권고 2: 미팅 빈도를 격주로 바꿔라 (위 3-2 참조)

월 1회 풀미팅 대신 격주 1회 (풀 + 카톡 follow-up). 부하는 거의 동일, 학습 속도 2배. 사장님과의 관계도 더 끈적해진다.

### 권고 3: "리포트 자가 검증"을 외주화하라

이승훈이 자기 리포트를 자가 검증하면 안 된다. 박승현 또는 엄수빈에게 **15분만** 보여주고 "이거 무슨 말인지 한 줄로 말해봐"를 시켜라. 한 줄로 못 말하면 L1 실패 확정 → 사장님 만나기 전에 수정. 이 한 단계가 6사이클 중 2~3사이클을 살린다.

## 9. ROI 한 줄

**ROI 계산이 의미 없는 단계다.** v0.1은 매출이 목적이 아니라 **"AI 제안 언어가 닿는가"라는 1개 가설**의 검증이 목적. 6사이클(약 6개월)에 60~100시간 투입해서 그 답을 얻으면 **ROI 무한대**(다음 사업 방향이 결정되므로). 못 얻으면 0.

단, 60~100시간을 던질 가치가 있는가는 다른 질문이다. 그 답은 기준 영업·운영이 무너지지 않는다는 전제에서만 YES.
