## 0. 한 줄 요약

> **6명 전원 Conditional-Go. v5 방향은 옳다. 단 MVP v0.1을 5가지 수정사항을 반영한 v0.1.1로 업데이트한 뒤 착수하는 것이 전원 권고.**

**No-Go 없음. 무조건 Go 없음.** 전원이 "방향은 맞지만 이 조건들이 빠지면 위험"이라는 같은 결론에 도달함.

## 2. 공통 합의 (6명 중 5명 이상 일치)

### 합의 1. v5 문제 정의는 유효하다
- 4/6 "슈퍼앱" 방향을 폐기하고 "AI 전략 참모"로 좁힌 것은 옳은 방향
- 건물주-임차인 프레임 폐기에 전원 동의 (미애의 4/6 카피 폐기 수용)

### 합의 2. MVP v0.1 그대로 착수하면 안 된다
- 전원이 최소 1가지 수정사항 제시
- No-Go는 없지만 "지금 상태 그대로 Go"도 없음

### 합의 3. 수동이 정답
- 진희, 미영, 주희 모두 "v0.1은 자동화 시도 금지"
- 진희: 자동화는 v0.3 이후
- 미영: 병목이 수동 수집에 있지만 자동화가 더 큰 부하
- 주희: 네이버는 사장님이 직접 다운로드 (스크래핑 법적 위험)

### 합의 4. 스펙에 측정 프로세스가 빠져 있다
- 미영: "L1~L3 측정 프로세스 자체가 스펙에 빠짐"
- 철수: "L4(지불 의향) 추가 필요"
- 영희: "L3 1회는 합격선 아닌 v0.2 진입 트리거로 재정의"

### 합의 5. 법·보안 안전망 먼저
- 주희·진희 모두 "LLM에 데이터 넣기 전 마스킹 게이트"
- 개인정보(전화번호) 해시화는 전원 동의

## 3. 의견 차이 (결론에 영향 있는 것)

### 이견 1. 대봉을 v0.1에 넣을 것인가?

| 입장 | 주장 | 주장자 |
|---|---|---|
| 유료 컨설팅으로 병행 | 회당 100~200만원 유료로 팔아라. v5의 "유료 10곳" 조건 검증 | 철수 |
| 30분 언어 인터뷰만 | N=1.5 효과로 언어 검증. 돈 받지 말 것 | 영희 |
| 미식가 먼저, 대봉 2차 | 원래 MVP 스펙 유지 | 승훈 원안 |

**김비서 판단:** **영희 안 채택.** 철수의 유료 컨설팅 안은 매력적이지만 v0.1에서 "AI가 낸 리포트의 품질"이 아직 검증 안 된 상태에서 돈을 받는 건 **(주희의 법적 리스크 확장)** + **(미애의 "첫 미팅 금지어 6개" 환경에서 돈 얘기는 조기 신뢰 붕괴)**. 다만 철수의 "유료 의향"은 **L4 지표로 변환**해서 v0.1.1에 포함 (돈 안 받고 "만약 이걸 매월 2~3만원에 받으신다면 어떠세요?" 반응만 수집).

### 이견 2. 데이터 소스 축소?

| 입장 | 주장 | 주장자 |
|---|---|---|
| 4개 유지 | 원래 스펙 (KT + 신한카드 + OKPOS + 네이버 예약) | 승훈 원안 |
| 2개로 축소 | 첫 사이클은 OKPOS + KT만. 신한카드/네이버 예약은 2사이클부터 | 미영 |
| 매칭률 실측 먼저 | 사이클 시작 전 0.5인일로 OKPOS×네이버 예약 매칭률 실측 | 영희 |

**김비서 판단:** **영희+미영 절충.** 첫 사이클(Cycle 1)은 **OKPOS + KT로 시작**, 동시에 **매칭률 실측(0.5인일)**을 병행. 실측에서 매칭률 30% 이상 나오면 Cycle 2부터 네이버 예약 추가. 신한카드는 Cycle 3 이후. 이유: 미영의 "빠른 속도" + 영희의 "데이터 정합성" 둘 다 잡음.

### 이견 3. 미팅 주기?

| 입장 | 주장 | 주장자 |
|---|---|---|
| 월 1회 | 원래 스펙 | 승훈 원안 |
| 격주 | 피드백 루프 끊김 방지 | 미영 |

**김비서 판단:** **월 1회 유지.** 미영의 격주 안은 부하를 줄이는 게 아니라 늘림 (월 사이클 16시간이 격주 32시간으로). 대신 **미팅과 미팅 사이에 카톡으로 한 번 중간 체크** (사장님이 액션 1개 실행했는지만 확인).

## 4. 최종 권고안 — MVP v0.1.1 (수정 버전)

원래 v0.1 스펙에 **5가지 수정사항**을 반영한 v0.1.1로 착수.

### 수정사항 1. [미영+영희] 데이터는 2개로 시작 + 매칭률 실측 병행

**변경 전:** KT + 신한카드 + OKPOS + 네이버 예약 (4개)
**변경 후:**
- **Cycle 1 (첫 리포트):** OKPOS + KT 2개만
- **병행 실측:** 승훈이 0.5인일 써서 OKPOS×네이버 예약 매칭률 실측 (과거 3개월치)
- 매칭률 ≥30%: Cycle 2부터 네이버 예약 추가
- 매칭률 <30%: 네이버 예약 사용 방식 재설계
- 신한카드: Cycle 3 (Cycle 2 안정화 후)

**이유:** 단순하게 시작 → 병목 실측 → 단계적 확장. 미영의 속도 + 영희의 정합성 동시 확보.

---

### 수정사항 2. [미영] L1~L3 측정 프로세스 스펙 추가

**빠진 것:**
- 미팅에서 L1/L2/L3을 어떻게 기록할 것인가?
- 승훈 기억에만 의존하면 피드백 루프가 실패함

**추가해야 할 것 (v0.1.1 스펙에 명시):**
- **미팅 노트 템플릿** (A4 1장, 체크리스트 형식)
  - L1 통과 여부 (사장님 질문 수, 반응)
  - L2 합의한 액션 (원문 그대로)
  - L3 직전 미팅에서 합의한 액션의 실행 여부
- **녹음** (사장님 동의 전제 — 주희의 합의서에 포함)
- **알파 독자 1명** — 리포트를 사장님 보여주기 전에 "이거 이해되세요?" 먼저 물어볼 사람 (승훈 지인 소상공인 1명)

---

### 수정사항 3. [주희+진희] 법·보안 안전망 5조건 (선행 작업)

**v0.1.1 착수 전 반드시 완료해야 할 것:**

1. **미식가 사장님과 1장 관계분리 합의서**
   - 건물주 지위와 자문자 지위를 분리 명시
   - 이 자문 관계가 임대차 조건·갱신에 영향 없음을 명시
   - 무상 자문임을 명시
   - 데이터 사용 범위(내부 분석 한정) 명시

2. **개인정보 처리위탁 계약서** (간이)
   - 전화번호 처리 근거 확보
   - 보관 기간, 파기 조건

3. **전화번호 즉시 해시화 파이프라인**
   - 네이버 예약 데이터 수집 후 **LLM에 넣기 전** 전화번호를 해시로 변환
   - 원본 전화번호는 슈파 시스템에 저장하지 않음

4. **외부 빅데이터 내부 한정 사용**
   - KT/신한카드 데이터는 **미식가 리포트 작성용으로만** 사용
   - 다른 가게에 재가공·재판매 금지 (v0.1에서는)

5. **네이버 예약은 사장님이 직접 다운로드**
   - 승훈이 사장님 계정으로 로그인하지 않음
   - 사장님이 네이버 예약 화면에서 엑셀/CSV 다운로드 → 승훈에게 전달
   - 스크래핑 금지

**이 5조건이 안 갖춰지면 v0.1.1 착수 금지.**

---

### 수정사항 4. [미애] 언어 재설계

**v0.1.1 스펙에 추가:**

**첫 미팅 첫 문장 (승훈이 꺼낼 말):**
> "사장님, 광고비는 매달 나가는데 왜 손님이 안 오는지는 아무도 안 알려주잖아요. 그거 한 달에 한 번 정리해 드리려고요."

**금지어 6개 (리포트 + 미팅):**
1. AI
2. 빅데이터
3. 세그먼트
4. 타겟팅
5. 전환율
6. 인사이트

**대체어:**
- 타겟 → "어떤 손님"
- 세그먼트 → "어떤 분들"
- 인사이트 → "뭐가 보이냐면"
- 전환율 → "몇 명이 진짜 오시는지"

**리포트 톤 가이드:** "동네 30년 형님 + 백종원 솔루션"
- 존댓말 유지, but 친근함
- 숫자는 쓰되 "이게 뭐냐면"으로 풀어서 설명
- "결론: X"보다 "제가 보기엔 X입니다"
- 액션은 "이거 해보세요"가 아니라 "이거 한번 해봐도 좋을 것 같아요"

**슈파 이름:** 유지. 단 **미식가 첫 미팅에선 이름을 거의 꺼내지 않음**. 사장님이 사는 건 이름이 아니라 "승훈씨가 한 달에 한 번 들고 오는 그 종이".

---

### 수정사항 5. [철수] 성공 기준에 L4 추가 (지불 의향, 비수금)

**변경 전:** L1(이해) → L2(동의) → L3(실행). L3 1회 = 합격.
**변경 후:** L1 → L2 → L3 → **L4(지불 의향 질문)**

- L3 도달한 후, 승훈이 다음 미팅에서 **"만약 이걸 매월 2~3만원에 받으실 수 있다면 어떠세요?"** 질문
- 사장님 반응을 3단계로 기록: 긍정("그거면 받고 싶다") / 중립("글쎄") / 부정("돈 들면 싫다")
- **돈을 실제로 받지 않음.** 반응만 수집.

**이유:**
- 철수: "검증 지표가 후퇴했다"에 대한 최소 응답
- 주희: 돈을 실제로 받으면 계약 관계가 생김 → 법적 리스크 급상승
- 영희: L3 1회는 v0.2 진입 트리거로 재정의 → L4 반응이 긍정이면 v0.2 확장, 중립/부정이면 v0.1.1 재설계

**합격선 (v0.1.1 최종):**
- **v0.2 진입 조건 = L3 1회 달성 + L4 긍정 1회**
- L3만 있고 L4 부정 → 제안 방식은 맞으나 유료 전환 불가 → 수익 모델 재설계
- L3 못 감 → 제안 언어 자체 재설계

## 6. 수정된 MVP 스펙 요약 (v0.1.1)

| 항목 | v0.1 (원안) | v0.1.1 (권고) |
|---|---|---|
| 외부 데이터 | KT + 신한카드 | Cycle 1: KT만 / Cycle 3부터 신한카드 |
| 내부 데이터 | OKPOS + 네이버 예약 | Cycle 1: OKPOS만 + 매칭률 실측 병행 / Cycle 2부터 네이버 예약 조건부 |
| 전달 방식 | 직접 미팅 → 카톡 | 직접 미팅 (월 1회) + 카톡 중간 체크 |
| AI 깊이 | 깊이 2 | 깊이 2 + **금지어 6개** + **톤 가이드** |
