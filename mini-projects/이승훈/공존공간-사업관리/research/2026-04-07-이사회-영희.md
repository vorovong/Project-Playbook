> 작성일: 2026-04-07
> 비교 기준: 2026-04-06 1라운드 영희 브리핑

## 0. 결론 (먼저)

> **수정 필요 (조건부 진행).**
>
> v5 문제 정의는 1라운드 대비 **선명해졌고 데이터 관점에서 더 방어 가능**하다. 그러나 MVP v0.1의 "표본 1, L3 1회 = 성공" 기준은 **통계적으로 무의미**하다. 통계적 검증이 아니라 **언어/UX 검증**이라고 명시적으로 재라벨링하면 v0.1은 진행 가능하다. 단, 그 즉시 v0.2에서 N≥5를 약속해야 한다.

| 판단 항목 | 4/6 영희 | 4/7 영희 |
|---|---|---|
| 문제 정의의 선명도 | 모호 ("플랫폼") | 선명 ("AI 전략 참모") |
| 시장 차별화 | 양면 데이터 | "숫자→전략" 공백 — **더 강함** |
| 표본 N | 3 → 5~10 필요 | 1 → **여전히 부족** |
| 추천 결정 | 도구로 시작, 5~10동 확보 | **언어 검증으로 재라벨링하라** |

## 1. 4/6 대비 입장 변화

**바뀐 것:**
1. v5 프레이밍("전략적 답을 내주는 사람이 없다")은 4/6에 영희가 지적한 "캐시노트가 못 가는 레이어" 가설을 **이론적으로 더 잘 메운다.** 4/6에는 "양면 데이터"를 차별점으로 잡았는데, v5는 "해석·통합·전략화" 공백을 잡았다 — 후자가 더 방어 가능한 해자다.
2. MVP를 "데이터 양"이 아니라 "AI 제안 언어"로 재정의한 것은 **검증 비용 측면에서 합리적**이다. 4/6 시점에 영희가 우려한 "검증 전 확장" 위험을 v0.1이 구조적으로 회피한다.

**바뀌지 않은 것:**
1. **N=1은 N=3보다 더 나쁘다.** 4/6에 "N=3은 통계적으로 무의미"라고 했는데, v0.1은 N=1이다. 통계적 입장에서는 **악화**됐다.
2. "L3 1회 도달 = 성공"에는 통계적 근거가 0이다. 1회 성공은 우연과 구분 불가하다.

→ 결론: **v5 가설은 강화, v0.1 실험 설계는 약화.** 두 개를 분리해서 평가해야 한다.

### 2-5. 종합 — "이 상권 타겟은 OOO다"라는 답이 나오는가?

**답: 부분적으로 나온다.**

- KT(누가 있나) × 신한카드(얼마 쓰나) × OKPOS(우리에게 얼마 벌었나) × 네이버 예약(언제 누가 결정했나) 4개 조합으로 **세그먼트 가설**은 만들 수 있다.
- 그러나 "이 가설이 맞다"는 검증은 **인과 추론이 아니라 사장님의 직관 일치도**로 대체된다. 슈파 v0.1은 통계적 검증 도구가 아니라 **가설 생성기**다.
- 이걸 명시적으로 인정하면 v0.1 진행 OK. 통계적 답으로 포장하면 위험.

## 3. 표본 1 / L3 1회 성공 기준의 통계적 평가

**팩트:**

| 지표 | 수치 |
|---|---|
| v0.1 표본 매장 | 1 |
| v0.1 성공 기준 | L3 도달 1회 |
| 통계적 유의수준(α=0.05)에 필요한 최소 N | ≥30 [일반 룰] |
| 4/6 영희 권고 N | 5~10동 |
| v0.1 실제 N | 1 |

**해석:**
- N=1, 시행 1회의 성공률은 표본 크기 1의 베르누이 시행이다. **신뢰구간은 [0%, 97.5%]**다. 이걸로는 어떤 결론도 못 낸다.
- 다만 v0.1 스펙 문서가 명시하듯 검증 대상이 "AI 제안 언어"라면 이는 **정량 검증이 아니라 정성 검증**이다. 사용성 테스트(UX research) 분야에서 N=5는 사용성 문제의 80%를 발견한다는 닐슨 룰이 있다. **이 기준이라면 N=1은 여전히 부족하다(80% 발견을 위한 최소 N=5).**

**결론:**
- "L3 1회 = MVP 합격" 표현은 **통계적 주장이 아니라 진행 결정 트리거**로 다시 라벨링해야 한다.
- 정확한 표현: "L3 1회 도달 시 v0.2 진입을 검토한다. 단, v0.2는 N≥5를 전제로 한다."
- 4/6의 "5~10동" 권고는 v5 MVP에서도 **여전히 유효**하다. 단지 v0.1이 아니라 v0.2의 합격선으로 옮겨야 한다.

## 4. 매칭 로직(날짜+시간+인원) 정합성 리스크

### 4-1. 매칭률 정량 추정

| 가정 | 값 [추정] |
|---|---|
| 미식가 예약 비중 | 50% |
| 예약 시간 ±15분 내 OKPOS 주문 발생 비율 | 80% |
| 인원수 정확 일치 비율 (예약 4명 → 주문 4인 세팅) | 70% |
| **최종 매칭률** (예약 손님 중) | **0.5 × 0.8 × 0.7 ≈ 28%** |
| **전체 손님 중 매칭률** | **0.5 × 28% ≈ 14%** |

**해석:**
- 전체 매출 데이터의 ~14%만 "사람 × 매출" 짝을 갖춘다 [추정].
- 이 14%가 워크인을 대표한다고 보기 어렵다 → **세그먼트 분석 시 표본 편향(예약 가능한 그룹 위주)**이 들어간다.
- 매칭 임계 완화(±30분, 인원±1명 허용)로 매칭률은 30~40%까지 올릴 수 있으나 false positive 증가.

### 4-2. v0.1이 무너지는가?

**아니다, 다만 조건부.**

- v0.1의 검증 대상이 "리포트 언어"라면 14% 표본으로도 정성적 시그널은 만들 수 있다.
- v0.1이 **"이 타겟이 맞다"는 정량 주장**을 하려 하면 14%로는 무너진다.
- **권고: 리포트 본문에 "샘플 매칭률 X%, 분석은 직관 검증용"이라고 명시할 것.** 사장님에게도 솔직하게.

## 5. 분기/반기 측정 단위와 데이터 주기 정합성

**v5 가설:** 목적형 약속상권은 분기/반기 단위로 봐야 의미 있음.

**데이터 주기:**
- KT 유동인구: 월 단위 → 분기 = 3개 데이터포인트.
- 신한카드: 월 단위 → 분기 = 3개.
- OKPOS: 일 단위 → 분기 = ~90개.
- 네이버 예약: 건 단위 → 분기 = 수백 건.

**판정:**
- 외부 데이터(KT/신한)는 **분기당 3개 점**으로 분기 단위 트렌드 1개를 만든다. 통계적 비교는 불가, 방향성 관찰만 가능.
- "분기 단위 결론"이 의미하는 바를 정확히 할 필요가 있다 — **변화 감지가 아니라 평균 프로파일링**이라면 가능. **전후 비교 검증**이라면 최소 4분기(1년) 데이터가 필요하다.
- v0.1 1~3개월 안에 분기 단위 결론을 내겠다는 것은 **데이터가 도착하기 전에 결론을 내는 셈**이다.

**권고:** v0.1 리포트에서 "분기/반기 변화 추이"는 다루지 않는다. 최초 리포트는 **현재 시점 스냅샷 프로파일링**으로 한정한다. 추세 분석은 v0.2 이후, 최소 6개월 데이터 축적 후로 미룬다.

## 6. 데이터 관점 수정 권고

### 권고 1 (강): v0.1 라벨링 변경

| 현재 표현 | 권고 표현 |
|---|---|
| "MVP 합격선: L3 1회 도달" | "v0.2 진입 트리거: L3 1회 + 사장님 정성 피드백 3건 이상" |
| "AI 전략 참모 검증" | "AI 제안 **언어/포맷** 검증 (정량 검증은 v0.2)" |
| "이 타겟을 노리세요" | "이 타겟 가설이 사장님 직관과 일치하는지 확인하세요" |

이유: 통계적 주장과 정성적 학습을 섞으면 자기기만이 발생한다.

### 권고 2 (중): 미식가 + 대봉 동시 시작 vs 순차

**옵션 A (스펙 그대로):** 미식가만 → L3 후 대봉.
- 장점: 자원 집중, 빠른 1회차 학습.
- 단점: N=1, 미식가 사장님 개인 특성에 학습이 오염될 위험.

**옵션 B (영희 권고 — 약):** 미식가 우선이되, **대봉 사장님과 30분 인터뷰 1회를 v0.1 기간 내 병행.**
- 비용: 미팅 1회 추가 (~3시간).
- 효과: "미식가 사장님이 이해하는 언어"가 "다른 목적형 약속상권 사장님에게도 통하는지" 1회 교차 확인 → N=1.5의 효과.
- 정식 L1~L3 사이클은 미식가에 집중. 대봉은 "언어만" 검증.

**권고: 옵션 B.** 자원 부담 대비 학습 가치가 크다.

### 권고 3 (강): 네이버 예약 데이터 접근 방법을 v0.1 착수 전 확정

스펙 4번 오픈 이슈에 있지만, 이게 **v0.1 전체의 단일 실패 지점**이다. API 없으면 스크래핑 → 합법성/신뢰성/지속성 모두 리스크. **이 결정이 안 나면 v0.1은 0주차에 막힌다.**

### 권고 4 (약): 매칭률을 v0.1 첫 번째 산출물로

리포트 작성 전에 **"OKPOS × 네이버 예약 매칭률 측정"** 자체를 첫 번째 데이터 산출물로 만든다. 14% [추정]가 실제로 어떤 값인지 모르면 이후 모든 분석이 모래 위에 있다. **0.5인일 작업**. ROI 매우 높음.
