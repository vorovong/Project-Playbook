## 2. 4개 데이터 소스별 수집 난이도 평가

### 2-1. KT 유동인구 — 난이도 중 / 자동화 어려움

- **현실:** KT 빅사이트(BigSight)는 웹 포털에서 지역·기간·인구속성을 골라 **엑셀/CSV로 다운로드**하는 구조. 공식 외부 API는 일반 사용자에게 열려 있지 않다.
- **v0.1 권고:** **수동 다운로드 OK.** 한 달에 한 번, 승훈이 직접 행궁동 데이터를 받아서 폴더에 떨군다. Puppeteer 자동 로그인은 가능하지만 인증·캡차·약관 변경 리스크가 있어 "v0.1 가성비 마이너스".
- **자동화 가능성:** v0.2~0.3에서 Puppeteer 스크립트화 가능. 단, 캡차 떴을 때 우회하지 말고 알림 보내고 멈추는 설계가 안전.

### 2-2. 신한카드 빅데이터 — 난이도 중 / 자동화 어려움

- **현실:** 신한카드 빅데이터 플랫폼도 KT와 마찬가지로 **포털 → 리포트 다운로드** 모델. API는 B2B 기업 계약 제공.
- **v0.1 권고:** **수동 다운로드 OK.** KT와 같은 흐름.
- **주의:** 라이선스/이용약관에 "재배포 금지" 조항이 있는지 확인 필요. 사장님께 보여주는 리포트에 "신한카드 데이터 기반"이라고 명시할지, 가공된 인사이트만 줄지 결정해야 한다. **데이터 보안 체크 필수.**

### 2-3. OKPOS — 난이도 낮음 / 이미 작동

- **현실:** `okpos-sales.js`가 이미 작동 중. iframe + IBSheet 크롤링 패턴 확립.
- **v0.1 권고:** **재활용 100%.** 슈파 입력으로 그대로 쓴다. 새 코드 0줄.
- **유일한 리스크:** OKPOS UI가 바뀌면 깨진다. 4/6에 제안한 "조용한 실패 알림" 제안을 여기서 다시 강조한다 — 슈파 리포트 생성 직전에 "OKPOS 데이터가 최신인가?" 체크 한 줄.

### 2-4. 네이버 예약 — 난이도 중상 / **새 작업 필요**

- **현실:**
  - 공식 API: 네이버 예약은 **파트너 API**가 있지만 일반 사장님 계정으로는 열리지 않는다 (가맹 파트너십 필요).
  - 사장님 페이지(스마트플레이스 사장님 모드): 로그인 후 예약 목록을 **CSV 내보내기** 가능. 이걸 활용하는 게 v0.1에 가장 빠르다.
  - 스크래핑: Puppeteer로 가능하나 네이버 로그인 보안(2FA, 캡차)이 까다롭다.
- **v0.1 권고:** **수동 CSV 내보내기 → 폴더 떨구기.** 자동화 시도하지 말 것. 월 1회 미팅 주기에 "사장님 만나기 전날 승훈이 5분 작업"이면 충분하다.
- **자동화는 v0.3 이후:** 사장님 권한을 받고 Puppeteer 로그인 자동화 — 단, 네이버 약관 위반 가능성 검토 필수.

### 종합

| 소스 | v0.1 방법 | 자동화 적기 |
|---|---|---|
| KT 유동인구 | 수동 다운로드 | v0.3 |
| 신한카드 | 수동 다운로드 | v0.3 |
| OKPOS | 자동 (재활용) | 이미 됨 |
| 네이버 예약 | 수동 CSV | v0.3 |

**핵심:** v0.1은 "수동 데이터 + 자동 리포트". 이게 가장 가성비 좋다. 자동화 욕심을 내면 검증이 2개월 늦어진다.

## 3. AI 전략 레이어 — 가장 큰 위험과 완화책

### 3-1. 위험: AI가 깊이 있는 답을 낼 수 있는가?

비유하자면, **요리사에게 "냉장고에 뭐 있는지" 알려주는 것과 "오늘 저녁에 뭘 만들지" 정해주는 건 다른 일이다.** 현재 LLM(Claude Opus 4.6 등)은 전자에는 능숙하지만 후자는 **프롬프트 설계와 컨텍스트 품질에 따라 결과가 크게 갈린다.**

**위험 시나리오 3가지:**

1. **일반론 함정** — "20~30대 여성 타겟이 유망합니다. 인스타그램 마케팅을 강화하세요." (어디서 본 것 같은 답)
2. **숫자 환각** — KT 데이터에 없는 수치를 그럴듯하게 만들어 냄. ("행궁동 30대 여성 유동인구 32% → AI가 지어낸 숫자")
3. **근거-결론 단절** — 결론은 그럴듯한데, 근거 데이터를 보면 결론과 무관함

### 3-2. 완화책 (반드시 v0.1에 넣을 것)

**완화 1: 데이터를 "숫자 그대로" 프롬프트에 박는다 (Grounding)**
- 4개 소스에서 뽑은 핵심 표를 **JSON 또는 마크다운 표** 형태로 프롬프트에 그대로 넣는다.
- "이 표 안의 숫자만 사용하라. 표에 없는 수치는 절대 인용하지 마라"는 시스템 프롬프트.
- 효과: 환각 확률 80% 감소.

**완화 2: 2단계 생성 (Draft → Critic)**
- 1단계: AI가 리포트 초안 작성
- 2단계: 같은 AI에게 "이 리포트의 모든 숫자가 입력 데이터에 실제 존재하는지 검증하라"고 자기 검토
- 검증 실패하면 1단계 재실행
- 비용은 2배 들지만, 환각 잡는 가장 확실한 방법

**완화 3: 승훈의 사전 자가 검증 게이트**
- 리포트 생성 후 **사장님 만나기 전에 승훈이 먼저 읽고 OK 사인.** 이게 v0.1 스펙에도 이미 있는 걸 알지만, 명문화한다 — "승훈이 OK 안 한 리포트는 절대 미팅에 들고 가지 않는다." 이게 v0.1의 안전장치다.

**완화 4: 프롬프트 버전 관리**
- 매 리포트마다 어떤 프롬프트 버전이 생성했는지 기록 (`prompts/v0.1.0.md`, `v0.1.1.md` ...)
- 리포트와 함께 저장. 나중에 "어떤 프롬프트가 잘 먹혔는지" 추적 가능.

### 3-3. 결론

**위험은 실재한다. 그러나 완화 가능하다.** 현재 LLM 기술로 "한 매장 + 4개 데이터 + A4 1장"이라는 좁은 범위는 충분히 품질 있는 답을 낼 수 있다. **단, 그냥 "이 데이터로 리포트 써줘"는 100% 실패한다.** 위 4개 완화책 없이 시작하면 안 된다.

## 4. v0.1 → v0.2 확장 시 기술 부채

수동 → 자동 전환의 리팩토링 비용은 **낮음**. 이유:

1. **데이터 형식만 통일하면 끝.** v0.1에서 수동 다운로드한 KT/신한카드/네이버 CSV를 정해진 폴더에 정해진 파일명으로 떨구는 규칙만 정해두면, v0.2의 자동화가 같은 폴더에 같은 파일을 만들면 된다. AI 파이프라인은 손대지 않아도 됨.
2. **AI 레이어는 데이터 소스를 모른다.** AI는 "JSON/표"만 본다. 그게 사람이 만든 거든 자동화가 만든 거든 상관없다. 이 분리만 지키면 부채 없음.
3. **권고 폴더 구조:**
   ```
   슈파/
     data-input/
       2026-04/
         kt-yudongin.csv
         shinhan-card.csv
         okpos-sales.json   ← 자동 (이미 있음)
         naver-reservation.csv
     reports/
       2026-04-미식가-v0.1.0.md
     prompts/
       v0.1.0.md
   ```
4. **유일한 부채 가능성:** 프롬프트가 "사람이 정리한 표 형식"에 너무 맞춰져 있으면, 자동화가 만든 raw 데이터에서 다시 정리 단계가 필요. **해결책:** v0.1에서도 "표 정리 단계"를 별도 함수로 분리해두기.

**총평:** v0.1을 잘 설계하면 v0.2 전환 비용은 1~2일이면 충분.

## 5. v0.1 스펙에 추가 권고 (기술 관점)

스펙은 이미 잘 짜여 있다. 기술 관점에서 5가지만 보강 제안.

1. **프롬프트 로그 보존** — 리포트마다 "어떤 프롬프트 + 어떤 입력 데이터 + 어떤 출력"을 한 세트로 저장. 나중에 디버깅·개선의 유일한 단서.
2. **AB 변형 2개 생성** — 같은 데이터로 프롬프트 약간 다른 2개를 돌려 2가지 리포트를 생성. 승훈이 둘 중 더 나은 걸 골라 미팅에 가져감. 자연스러운 AB 테스트가 누적된다. (비용: LLM 호출 2배 = 무시할 수준)
3. **사장님 피드백 구조화** — 미팅 후 "L1/L2/L3 어디까지 갔나, 왜 막혔나"를 정해진 포맷으로 한 페이지 기록. 4번 미팅 후 패턴이 보임.
4. **데이터 출처·시점 명시 의무** — 리포트 하단에 항상 "근거: KT 2026-03, 신한카드 2026-Q1, OKPOS 2026-03-01~31, 네이버 예약 2026-03" 표기. 사장님이 "이거 언제 데이터예요?"라고 물을 때 답할 수 있어야 한다. **블랙박스 최소화 원칙.**
5. **개인정보 마스킹 게이트** — 네이버 예약 데이터에 전화번호가 있다. AI 프롬프트에 보내기 전에 **전화번호는 해시 처리하거나 제거.** 사장님 한 명 데이터라도 LLM 외부 API로 보내는 순간 보안 검토 대상이다. **데이터 보안 준수 원칙.**

## 6. MVP v0.1 기술 구현 순서 제안

기존 스펙 §4의 빌드 순서에 살을 붙인 버전:

| 단계 | 작업 | 예상 시간 | 의존성 |
|---|---|---|---|
| 1 | 폴더 구조 + 파일명 규칙 확정 | 30분 | — |
| 2 | OKPOS 출력 → 슈파 입력 형식 변환 함수 | 1시간 | 1 |
| 3 | KT/신한카드/네이버 수동 다운로드 SOP 작성 (스크린샷 포함) | 1시간 | 1 |
| 4 | 데이터 → AI 입력용 표 정리 함수 (개인정보 마스킹 포함) | 2시간 | 2,3 |
| 5 | **프롬프트 v0.1.0 작성** (가장 중요. 여러 번 고친다) | 3~5시간 | 4 |
| 6 | LLM 호출 + 2단계 검증 (Draft→Critic) | 2시간 | 5 |
| 7 | A4 마크다운 → PDF 변환 (Puppeteer 재활용) | 1시간 | 6 |
| 8 | **승훈 자가 검증 1회** (실데이터로 끝까지 돌려보기) | 2시간 | 7 |
| 9 | 프롬프트 튜닝 (자가 검증 결과 반영) | 2~4시간 | 8 |
| 10 | 첫 미팅 진행 + 피드백 기록 | — | 9 |

**총 구현 시간: 약 15~20시간 (= 2~3일 집중 작업).** 4/6에 "MVP 2~4주"라 했는데, v0.1은 범위가 더 좁아서 **1주일 안에 첫 미팅 가능.** 4/6 평가보다 더 빠르다.

**가장 시간 많이 드는 곳:** 5번(프롬프트). 여기서 절약하면 안 된다.

## 7. 강점 / 약점 / 개선

### 강점 3가지

1. **위험이 한 곳에 압축됐다.** 4/6은 "여러 영역 동시 위험"이었는데, v0.1은 "프롬프트 품질" 한 점에 집중. 이게 검증 관점에서 가장 큰 진전.
2. **기존 자산 활용도 높음.** OKPOS 수집기, Puppeteer(PDF 변환), Node.js 환경이 이미 있다. **새로 배워야 할 기술 0개.**
3. **수동 → 자동 마이그레이션 부채 없음.** 폴더 구조 + JSON 형식만 통일하면 v0.2 자동화는 AI 레이어를 건드리지 않고 가능.

### 약점 3가지

1. **프롬프트가 깨지는 경계 케이스를 모른다.** 첫 5개 리포트는 거의 확실히 "이상한 답"을 낸다. 이걸 빠르게 발견·수정할 수 있는 평가 루프가 v0.1에 명시되어 있지 않다.
2. **데이터 보안 절차가 미정.** 네이버 예약 전화번호, 사장님 매출이 외부 LLM API로 나간다. 마스킹/온프레미스 LLM/계약 등의 결정이 필요한데 스펙에 없음.
3. **"AI가 못한다"가 확정될 때의 출구가 없다.** L1에서 막히면 "프롬프트 재설계"가 답이라 했는데, 3~4번 재설계해도 안 되면? 그때 슈파 가설을 폐기할지, 다른 접근(예: 사람 컨설턴트 + AI 보조)으로 갈지 미리 정해두면 좋다.

### 개선 제안 3가지

1. **프롬프트 평가 루브릭 1장 만들기 (1시간)** — 리포트가 나오면 ① 환각 여부, ② 결론-근거 정합성, ③ 액션의 구체성 3개를 5점 척도로 매기는 표. 승훈이 매 리포트 자가 검증할 때 사용. 평가가 객관화되면 프롬프트 개선 방향이 보인다.
2. **개인정보 마스킹 함수를 첫 코드로 작성** — 모든 데이터 흐름의 가장 앞단에 둔다. "마스킹 안 된 데이터는 AI로 못 간다"는 강제 게이트. 1시간 작업.
3. **3회 미팅 회고 미리 캘린더에 박기** — 미팅 3회 후 "프롬프트가 사장님 언어와 얼마나 가까워졌나"를 평가. 못 가까워졌으면 v0.1 폐기 또는 피벗 결정. 미리 박아둬야 매몰비용에 안 끌린다.

## 8. 최종 의견

> "기술로 더 잘할 수 있는가?"

**예. v0.1은 기술적 모험이 아니라 기술적 검증이다.** 새 기술을 배워서 만드는 게 아니라, 이미 있는 기술 부품(LLM API, OKPOS 수집기, Puppeteer)을 슈파 모양으로 조립하는 일이다.

**다만 "조립"의 가운데에 있는 한 조각 — 프롬프트 — 이 사실상 v0.1 전체의 운명을 결정한다.** 이걸 가볍게 보면 안 된다. "AI한테 던지면 알아서 답함"은 환상이다. **AI는 '무엇을' 결정하는 게 아니라 '어떻게'를 돕는다는 우리 원칙에 가장 충실해야 하는 단계가 바로 여기다.** 프롬프트는 "사장님이 받았을 때 사람이 쓴 것처럼 자연스럽고, 숫자는 데이터 그대로고, 액션은 내일 시도할 수 있는 것"이라는 좁은 통로를 통과해야 한다.

**판단:** Conditional Go.
**조건:** §3 완화책 4개 + §5 권고 5개 중 최소 1·4·5번(프롬프트 로그, 출처 명시, 개인정보 마스킹) 반드시 포함.
**예상 일정:** 1주 안에 첫 리포트, 2주 안에 첫 미팅, 4주 안에 L3 도달 시도.

사장님 미팅은 사람(승훈)이 직접 하는 v0.1의 설계는 우리 원칙("인간적 상호작용을 AI로 대체하지 않는다")과 완벽히 일치한다. 이 부분은 칭찬받아 마땅하다. AI는 사장님 옆에 앉지 않는다. AI는 승훈이 사장님 옆에 앉기 전에 머리 정리를 도와줄 뿐이다. **이게 슈파의 정의여야 한다.**
