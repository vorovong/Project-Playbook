# 월정산 자동화 청사진 — 상세편 (v3)

> 공존공간에서 구축한 월 운영현황 자동화 시스템을 다른 업체에 적용하기 위한 상세 가이드
> 기본편(v2)을 읽은 뒤 이 문서를 참고

---

## 1. 수익 모델 이해 (가장 중요)

### 왜 중요한가

정산 자동화의 핵심은 **"뭘 매출로 잡느냐"**다.
이걸 틀리면 모든 숫자가 틀린다.

### 공존공간 수익 모델

공존공간은 **건물주(임대업)**다. 직접 장사하는 게 아니라, 임차인이 벌어온 돈에서 수수료를 받는 구조.

```
미식가의주방 (임차인)
  총매출: 4,351만원 (이건 공존공간 매출이 아님!)
  → 카드 수수료 15% + 부가세 = 약 600만원 ← 이게 공존공간 매출
  → 현금 수수료 13% + 부가세 = 약 70만원 ← 이것도

포토인더박스 (임차인)
  총매출: 300만원
  → 운영수수료 15% = 45만원 ← 공존공간 매출
  → 임대료 55만원 ← 고정 수입

재생전술 (직영)
  총매출: 150만원 ← 이건 직영이라 전액 매출
```

### 핵심 공식

```
공존공간 매출 = 미식가_카드수수료총액 + 미식가_현금수수료총액
             + 포토인더박스_총매출
             + 재생전술_총매출

※ 임대료수익(763만)은 매출 집계에서 별도 처리 (고정 수입)
```

### 새 업체 적용 시

| 질문 | 왜 중요한가 |
|---|---|
| 직영인가 임차인인가? | 직영 = 총매출이 매출, 임차인 = 수수료만 매출 |
| 수수료 구조는? | 카드/현금 다를 수 있음, 부가세 포함 여부 |
| 임대료는 고정인가 변동인가? | 고정이면 CONFIG에, 변동이면 collector에 |

## 2. 주차 계산 로직 (getMonthRange)

### 규칙

공존공간은 **월~일 단위 4주**를 한 달로 본다.

- 매월 1일이 속한 주의 **월요일**부터 시작
- 단, 1일이 **일요일**이면 다음 날(월요일)부터 시작
- 4주(28일) = 해당 월
- 29일~31일은 **다음 달로 이월**

### 실제 예시

```
2026년 3월:
  3월 1일 = 일요일
  → 시작: 3월 2일 (월)
  → 종료: 3월 29일 (일)
  → 3월 30~31일은 4월로 이월

  1주차: 3/2(월) ~ 3/8(일)
  2주차: 3/9(월) ~ 3/15(일)
  3주차: 3/16(월) ~ 3/22(일)
  4주차: 3/23(월) ~ 3/29(일)

2026년 4월:
  4월 1일 = 수요일
  → 시작: 3월 30일 (월) ← 전월 이월분 포함!
  → 종료: 4월 26일 (일)

  1주차: 3/30(월) ~ 4/5(일)
  2주차: 4/6(월) ~ 4/12(일)
  3주차: 4/13(월) ~ 4/19(일)
  4주차: 4/20(월) ~ 4/26(일)
```

### 코드 (lib/weeks.js)

```javascript
function getMonthRange(month) {
  const [y, m] = month.split('-').map(Number);
  const first = new Date(y, m - 1, 1);
  const dow = first.getDay(); // 0=일, 1=월, ... 6=토

  // 1일이 일요일이면 +1, 나머지는 해당 주 월요일로
  const offset = dow === 0 ? 1 : -(dow - 1);
  const startDate = new Date(first.getTime() + offset * 86400000);
  const endDate = new Date(startDate.getTime() + 27 * 86400000);
  // ... 주차별 배열 생성
}
```

### 중요: 월 경계 쿼리

4월 1주차는 3/30부터 시작한다. 그래서 **매출 쿼리할 때 전월 데이터도 가져와야 한다.**

```javascript
// collectors.js에서
const months = new Set();
months.add(range.start.slice(0, 7)); // '2026-03' (이월분)
months.add(range.end.slice(0, 7));   // '2026-04'

// 두 달 모두 쿼리한 뒤, 날짜 범위로 필터
.filter(d => d.date >= range.start && d.date <= range.end)
```

## 5. Collector 상세 (데이터 수집 함수)

### collectors.js 구조

```javascript
module.exports = {
  collectors: [
    { key: 'misikga',   label: '미식가의주방 매출', fn: fetchMisikga,   needsRange: true },
    { key: 'utilities',  label: '공과금',           fn: fetchUtilities },
    { key: 'photo',      label: '포토인더박스',     fn: fetchPhoto },
    { key: 'purchases',  label: '구매내역',         fn: fetchPurchases },
    { key: 'classes',    label: '재생전술',         fn: fetchClasses },
  ],
};
```

### 각 collector가 하는 일

#### fetchMisikga (config, month, range) → needsRange: true

```
입력: month='2026-03', range={start:'2026-03-02', end:'2026-03-29', weeks:[...]}
처리:
  1. NOTION_SALES_DB에서 3월 + (필요시 2월) 데이터 쿼리
  2. range.start ~ range.end로 필터 (28일만)
  3. 주차별 그룹핑
  4. 수수료 계산: 카드 15% + 부가세 10%, 현금 13% + 부가세 10%
출력: {
  days: [...],              // 일별 데이터 배열
  weeks: { 1: {...}, 2: {...}, 3: {...}, 4: {...} },
  총매출, 실매출, 총할인, 영수건수합계,
  카드합계, 현금합계, 현금영수합계, 현금총,
  카드수수료, 카드수수료부가세, 카드수수료총액, 카드입금액,
  현금수수료, 현금수수료부가세, 현금수수료총액,
  수수료후입금,
}
```

#### fetchUtilities (config, month)

```
입력: month='2026-03'
처리:
  1. NOTION_UTILITY_DB에서 청구월='2026-03' 쿼리
  2. 종류별 분류 (전기/가스/수도)
  3. 전기는 지하/공용 분리
  4. 태양광: 전월 발전량 × 지하 전기 단가 = 절감액
  5. 정산율 적용 (2026-03까지 95%, 이후 100%)
출력: {
  가스비, 수도세, 전기세, 합계, rate, rateLabel,
  가스_고지서, 수도_고지서,
  전기_지하_고지서, 전기_지하, 전기_공용_고지서, 전기_공용,
  태양광절감액, 태양광발전량, 단가, 전기_지하_사용량,
}
```

#### fetchPhoto (config, month)

```
입력: month='2026-03'
처리: NOTION_PHOTO_DB에서 월='2026-03' 쿼리 → 부스별 카드/현금 + 재고/사용량
출력: { 부스: [{번호, 카드, 현금, 합계}, ...], 총매출, 재고: {...}, 사용: {...} }
```

#### fetchPurchases (config, month)

```
입력: month='2026-03'
처리: NOTION_PURCHASE_DB에서 3월 구매 쿼리 → 채널별/파트별 합산
출력: { items: [...], byChannel: {...}, byPart: {...}, 합계 }
```

#### fetchClasses (config, month)

```
입력: month='2026-03'
처리: NOTION_CLASS_DB에서 3월 클래스 쿼리 → 유형별 매출/비용/이익
출력: { classes: [...], 총매출, 총정산금액, 총재료비, 총강사비, 순이익 }
```

### 새 업체 collector 작성 팁

- `needsRange: true` → 주차 기반 월 계산이 필요한 경우 (POS 매출 등)
- `needsRange` 없으면 → 단순 월 필터 (공과금, 구매내역 등)
- collector는 반드시 `(config, month)` 또는 `(config, month, range)` 시그니처
- 리턴값의 key가 sheets.js에서 `data.misikga`, `data.utilities` 등으로 접근됨

## 6. Sheet 상세 (시트 빌더 함수)

### sheets.js 구조

```javascript
module.exports = {
  mainSheet: { build: build월결산 },    // 메인 페이지 (월 결산)
  subSheets: [                          // 서브페이지들
    { key: 'misikga-sales', title: (label) => `미식가의주방 (${label} 총매출)`, build: build미식가총매출 },
    { key: 'misikga-utility', title: () => '미식가의주방 공과금', build: build미식가공과금 },
    { key: 'photo', title: (label) => `포토인더박스 (${label})`, build: build포토인더박스 },
    { key: 'labor', title: () => '인건비', build: build인건비 },
    { key: 'purchase', title: () => '구매내역', build: build구매내역 },
    { key: 'class', title: () => '재생전술', build: build재생전술 },
  ],
};
```

### 모든 빌더 함수 시그니처

```javascript
function build시트명(config, month, data) → blocks[]
```

- `config`: config.js 값 전체
- `month`: '2026-03'
- `data`: collector 결과 전체 `{ misikga, utilities, photo, purchases, classes }`
- 리턴: 노션 블록 배열

### 시트 1: 월 결산 (메인 페이지)

## 9. 트러블슈팅 (겪었던 문제들)

### 문제 1: "매출이 구글시트랑 안 맞는다" (49.4M vs 43.5M)

**원인**: 구글시트는 30일(달력 기준), 자동화는 28일(4주 기준)
**해결**: 주차 기반 getMonthRange() 함수 도입. 29~31일은 다음 달로 이월.
**교훈**: 정산 기간 정의를 먼저 확인할 것.

### 문제 2: "수수료 계산이 틀렸다"

**원인**: 처음에 `수수료후입금`(미식가에게 돌아가는 돈)을 공존공간 매출로 잡음
**해결**: `카드수수료총액 + 현금수수료총액`(공존공간이 가져가는 수수료)으로 변경
**교훈**: 임차인 매출 ≠ 임대업 매출. "누구의 매출인가?" 반드시 확인.

### 문제 3: "포토인더박스 데이터 없으면 크래시"

**원인**: `photo.재고.필름.금액` 접근 시 photo가 빈 객체
**해결**: fetchPhoto에서 데이터 없으면 `{ 수량: 0, 단가: 0, 금액: 0 }` 형태의 빈 객체 리턴
**교훈**: 모든 collector는 빈 데이터 처리 필수.

### 문제 4: "노션 블록 100개 제한"

**원인**: 노션 API는 한 번에 children 100개까지만 허용
**해결**: `createSubPage()`에서 100개씩 나눠서 PATCH로 추가
**교훈**: 주차별 매출처럼 블록이 많은 시트는 자동 분할 필요.

### 문제 5: "페이지가 엉뚱한 곳에 생겼다"

**원인**: parentPageId를 미식가의주방 정산 페이지로 지정 (공존공간 바로 아래가 아니라)
**해결**: 노션 페이지 계층 확인 후 정확한 ID로 변경
**교훈**: parentPageId는 노션에서 직접 확인. 가정하지 말 것.

### 문제 6: "아카이브된 페이지에 DB 생성 실패"

**원인**: 이전에 쓰던 부모 페이지가 아카이브됨
**해결**: 활성 페이지의 ID를 다시 찾아서 지정
**교훈**: 노션 페이지 ID는 아카이브되면 사용 불가.

## 10. 자동 개발 루프 (Ralph Loop)

### 개요

새 업체 세팅이나 버그 수정을 자율적으로 돌릴 수 있다.

```bash
bash ralph.sh specs/작업.md 10
```

### 스펙 파일 작성법

```markdown
# [작업 제목]

## 목표
한 줄로 뭘 만들건지.

## 작업 내용
1. ...
2. ...

## 완료 기준
- [ ] 기준 1 (구체적, 자동 검증 가능)
- [ ] 기준 2
```

### 핵심: 완료 기준을 자동 검증 가능하게

| 좋은 기준 | 나쁜 기준 |
|---|---|
| `node xxx.js` 에러 없이 실행 | 코드가 잘 돌아감 |
| 노션에 페이지 생성됨 | 결과가 괜찮음 |
| `require('./file')` 성공 | 파일이 올바름 |
| 숫자가 43,512,515와 일치 | 숫자가 맞음 |

### Ralph Loop 실행 흐름

```
사용자: bash ralph.sh specs/newbiz.md 10

반복 1: Claude가 스펙 읽고 → 코드 작성 → 테스트 → 에러 발생 → RALPH_CONTINUE
반복 2: 에러 확인 → 수정 → 테스트 → 일부 통과 → RALPH_CONTINUE
반복 3: 나머지 수정 → 전체 통과 → RALPH_DONE

완료! (3회 반복)
```
