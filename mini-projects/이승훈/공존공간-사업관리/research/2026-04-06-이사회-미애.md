# 로하이사회 1라운드 -- 미애(내러티브 브랜드 디렉터) 브리핑

> 작성일: 2026-04-06
> 안건: "공존공간을 건물주-임차인 양면 슈퍼앱 플랫폼으로 확장할 것인가?"
> 렌즈: "사람들이 이걸 어떻게 느끼는가?"

---

## 0. 수신자 감정 선(先)판단

이 안건을 처음 접하는 사람이 느낄 첫 감정을 세 대상별로 먼저 짚겠다.

| 대상 | 첫 감정 | 왜? |
|------|---------|-----|
| **임차인** | "이거... 감시 아냐?" (경계 70% / 호기심 30%) | 건물주가 내 매출, 급여, 세무를 본다? 본능적 방어기제 |
| **건물주(투자자)** | "건물주가 왜 그걸 해줘?" (의아 60% / 흥미 40%) | 기존 건물주 역할 프레임에 안 맞음 |
| **일반 대중** | "건물주가 착한 척?" (냉소 50% / 감동 50%) | 한국 사회에서 '건물주'라는 단어 자체가 부정적 뉘앙스 |

**핵심 문제: '건물주'라는 단어가 한국에서 짊어진 감정적 부채가 크다.**
젠트리피케이션, 갑질, 임대료 폭등... 이 모든 이미지를 뚫고 "건물주가 당신을 돕겠다"는 메시지를 전달해야 한다. 이건 기능의 문제가 아니라 **서사의 문제**다.

## 1. 브랜드 포지셔닝 분석

### 현재 경쟁 지형에서 공존공간의 위치

```
                     [임차인 독립형]
                          |
          캐시노트 --------+-------- 토스플레이스
          (데이터)         |         (POS)
                          |
 [범용] ---------+--------+--------+--------- [업종특화]
                 |        |        |
             얼마에요      |      보스몬
             (회계)        |      (인력)
                          |
                          |
                   ?? 공존공간 ??
                          |
                     [건물주 연결형]
```

공존공간이 노리는 자리는 정말 **비어 있다**. "건물주-임차인 연결"이라는 축 자체가 존재하지 않는다.
이건 블루오션이 맞다. 문제는 -- 왜 비어 있었는지를 생각해야 한다.

**비어 있던 이유 3가지:**
1. 건물주 대부분이 "관리"에 관심이 없다 (수동 소득 추구)
2. 임차인은 건물주와 거리를 두고 싶어한다 (간섭 = 불편)
3. 두 집단의 이해관계가 충돌한다고 인식됨 (제로섬)

따라서 포지셔닝의 핵심은 **"이건 제로섬이 아니라 포지티브섬이다"를 감정적으로 납득시키는 것**이다.

### "감시 vs 도움" -- 임차인은 어떻게 느낄까?

임차인이 느끼는 감정은 **누가 먼저 제안하느냐**에 따라 갈린다.

| 시나리오 | 임차인 감정 |
|---------|------------|
| 건물주가 "이 시스템 쓰세요" | "감시당하는 느낌. 내 매출 들여다보려고?" |
| 임차인이 "이런 거 있으면 좋겠다" 하는데 건물주가 제공 | "오 건물주가 이런 것까지? 괜찮은데?" |
| 제3자 플랫폼인데 건물주가 비용을 대줌 | "무료면 일단 써볼까. 건물주한테 뭐가 보이는 건 아니겠지?" |

**결론: 임차인 데이터의 주권이 임차인에게 있다는 것을 구조적으로 보장해야 한다.**
건물주가 볼 수 있는 건 "공실 여부"와 "정산 관련 데이터"뿐이고, 세부 매출/급여/세무는 임차인만 볼 수 있다 -- 이 구조가 브랜드 메시지의 전제 조건이다.

## 2. 스토리 -- 어떤 서사로 포장해야 공감하는가?

### 쓰면 안 되는 스토리

| 스토리 | 왜 안 되는가 |
|--------|-------------|
| "착한 건물주" | 시혜적. "저는 다른 건물주와 달라요" -- 이건 자기PR이지 브랜드가 아니다 |
| "상생" (직접 사용) | 너무 많이 쓰여서 의미가 휘발됨. 정부 보도자료 느낌 |
| "건물주가 도와드립니다" | 갑이 을에게 베푸는 구도. 역효과 |

### 써야 하는 스토리: "같은 배를 탄 사람들"

**핵심 서사: 임차인이 망하면 건물주도 망한다.**

이건 미담이 아니라 **경제적 사실**이다. 공실은 건물주의 악몽이고, 임차인 매출이 올라야 임대료가 안정된다. "도와주는 것"이 아니라 "서로 연결된 운명을 인식하는 것"이다.

**서사 구조:**

```
[기존 프레임]  건물주 vs 임차인 (제로섬, 갑을관계)
     |
     v  전환점: "임차인이 폐업하면 누가 손해?"
     |
[새 프레임]   건물주 + 임차인 = 하나의 생태계 (포지티브섬)
     |
     v  공존공간이 이 생태계의 운영체제
```

**비유 하나:** 임대차 관계를 "집주인-세입자"가 아니라 **"공동 창업자"**로 리프레이밍.
건물주는 공간을, 임차인은 사업을 가져온다. 둘이 합쳐야 매출이 발생한다.
이건 과장이 아니다 -- 상가 임대의 본질이 원래 그렇다.

### 레퍼런스 스토리

- **WeWork 초기 서사 (망하기 전):** "커뮤니티로 세상을 바꾼다" -- 너무 거창해서 실패
- **Slice (피자집 플랫폼):** "우리는 피자집 편이다. 배달앱과 다르다" -- 적을 만들어서 성공
- **Toast:** "레스토랑 사람이 만든 레스토랑 도구" -- 내부자 관점으로 신뢰 획득

공존공간에 맞는 건 **Toast 모델**이다. "건물주가 임차인의 고충을 직접 겪어봤기 때문에 만든 시스템."

## 3. 네이밍 & 슬로건 제안

### 플랫폼 이름 후보

| 이름 | 연결고리 | 장점 | 리스크 |
|------|---------|------|--------|
| **공존** (그대로) | 브랜드 자산 그대로 활용 | 인지도 유지, 의미 직관적 | "건물 관리 회사" 이미지 고착 가능 |
| **공존 OS** | "운영체제"라는 메타포 | 기술 플랫폼 느낌, 확장성 | OS라는 용어가 임차인에게 낯설 수 있음 |
| **로하(LOHA)** | 모기업 브랜드 직접 사용 | 철학이 이름에 내재, 글로벌 확장 용이 | 인지도 0에서 시작 |
| **터전** | 한국어 감성, "사업의 터전" | 따뜻하고 직관적 | 확장 시 스케일감 부족 |
| **한지붕** | "한 지붕 아래" | 공동체 느낌, 기억하기 쉬움 | 다소 촌스러울 수 있음 |

**미애 추천: "공존" 유지 + 서브 브랜드 전략**

```
공존공간  =  물리적 공간 (건물 관리/운영)
공존 플랫폼  =  디지털 공간 (임차인 운영 도구)
```

"공존"이라는 두 글자가 이미 이 사업의 본질을 담고 있다. 버리기 아깝다.
건물에서 시작해서 디지털로 확장한다는 스토리 자체가 이 브랜드만의 고유한 여정이 된다.

### 슬로건 후보

| 슬로건 | 대상 | 톤 |
|--------|------|-----|
| **"한 건물이 하나의 팀이 됩니다"** | 전체 | 따뜻하고 비전 중심 |
| **"사장님, 혼자 하지 마세요"** | 임차인(소상공인) | 직접적, 공감형 |
| **"공실 없는 건물의 비밀"** | 건물주 | 실용적, 호기심 유발 |
| **"같은 주소, 같은 성장"** | 전체 | 연결감 강조 |
| **"건물이 사업을 돕는 시대"** | 투자자/언론 | 패러다임 제시 |

**미애 추천 1순위: "한 건물이 하나의 팀이 됩니다"**

이유: '건물주'라는 단어를 쓰지 않으면서도 건물주-임차인 관계를 재정의한다.
"팀"이라는 단어가 수직관계를 수평관계로 전환시킨다.
그리고 "됩니다"라는 현재형이 이미 일어나고 있는 일처럼 느끼게 만든다.

## 4. 타겟별 메시지 설계

### 4-1. 임차인(소상공인)에게

**핵심 감정:** "아 이거 나 혼자 다 해야 하는데 죽겠다"

```
[DO]
"사장님, 월급 계산하다 새벽 2시 된 적 있으시죠?
세금 신고 때마다 세무사한테 전화하기 민망하시죠?
그거, 저희가 해결해드립니다. 무료로."

[DON'T]
"건물주가 제공하는 통합 운영 플랫폼입니다."
(건물주라는 단어를 전면에 내세우지 않는다)
```

**메시지 전략:**
- 1단계 (진입): 건물주 언급 없이, **무료 운영 도구**로 접근
- 2단계 (정착): "이 건물에 입주하면 이런 혜택이 있다"로 자연스럽게 연결
- 3단계 (충성): "다른 건물로 이사 가면 이걸 못 쓴다" -- 자발적 락인

**킬러 카피:**
> "장사만 하세요. 나머지는 건물이 합니다."

이 한 줄이 모든 걸 담는다. "건물주"가 아니라 "건물"이 주어다. 공간 자체가 서비스를 제공하는 느낌. 사람 대 사람의 권력관계를 빼고, 공간이 알아서 돕는 이미지.

### 4-2. 건물주에게

**핵심 감정:** "공실 무섭다. 좋은 임차인 오래 있었으면 좋겠다"

```
"좋은 임차인은 저절로 오지 않습니다.
좋은 건물이 좋은 임차인을 만듭니다.

공존 플랫폼이 입주한 건물의 공실률: 0%
평균 임차 기간: 기존 대비 2.3배

건물의 가치는 임차인의 매출에서 나옵니다."
```

**메시지 전략:**
- 감성이 아닌 **숫자**로 설득 (공실률, 임차 기간, 건물 가치)
- "착한 건물주"가 아니라 **"똑똑한 건물주"** 프레이밍
- 비용이 아니라 **투자** 프레이밍 (임차인 지원 비용 = 공실 방지 보험)

### 4-3. 투자자에게

**핵심 감정:** "이거 시장이 있나? 확장 가능한가?"

```
"건물주-임차인 관계는 전 세계 상업 부동산의 기본 단위입니다.
하지만 이 관계를 디지털화한 플랫폼은 없습니다.

Square는 결제에서 시작해 운영 전체로 확장했습니다.
공존은 '공간'에서 시작해 운영 전체로 확장합니다.

Square의 진입점: 카드리더기
공존의 진입점: 임대차계약서

이미 임차인과의 관계가 존재합니다.
CAC(고객획득비용)가 0에 가깝습니다."
```

**메시지 전략:**
- "글로벌에 없는 모델" 강조 (진짜 없다)
- CAC 제로 = 최강의 유닛 이코노믹스 스토리
- Square/Toast 비유로 스케일 상상력 자극
- 단, "한국 상가 임대 시장 규모"를 숫자로 뒷받침해야 함

## 5. 브랜드 리스크 분석

### 위험 1: "건물주가 내 데이터를 본다" 공포

**심각도: 최상**

한국 소상공인에게 "내 매출을 건물주가 안다 = 임대료 올릴 근거를 준다"는 공식이 본능적으로 작동한다. 이건 합리적 우려다.

**대응:**
- 구조적 차단: 건물주가 볼 수 있는 데이터를 계약 단위로 제한 (정산 관련만)
- 임차인이 공유 범위를 직접 설정하는 대시보드
- "데이터 주권 헌장" 같은 상징적 선언 (법적 구속력 있는 약관)
- 브랜드 메시지에서 이 부분을 선제적으로 다루기 (숨기면 더 의심)

### 위험 2: "건물주 플랫폼"이라는 프레이밍 고착

**심각도: 상**

한 번 "건물주가 만든 플랫폼"으로 인식되면, 임차인 입장에서 "갑의 도구"로 느껴진다. 캐시노트나 토스플레이스처럼 "사장님 편"인 느낌이 아니라 "건물주 편"으로 인식될 위험.

**대응:**
- 론칭 초기부터 **임차인 커뮤니티**를 전면에 배치
- 임차인 성공 사례를 중심 콘텐츠로 (건물주 이야기는 뒤로)
- 임차인이 직접 추천하는 구조 (건물주가 깔아주는 게 아니라)

### 위험 3: "무료인데 뭔가 있겠지" 불신

**심각도: 중**

무료 서비스에 대한 한국 소비자의 학습된 경계심. "공짜면 내가 상품이다."

**대응:**
- 수익 모델을 투명하게 공개 ("건물주가 비용을 부담합니다. 왜? 공실보다 싸니까요")
- 또는 프리미엄 모델 도입 (기본 무료 + 고급 기능 유료)으로 비즈니스 모델 명확화

### 위험 4: 스케일 시 브랜드 희석

**심각도: 중 (장기)**

공존공간이라는 이름이 "특정 건물"의 느낌을 준다. 플랫폼이 100개 건물로 확장되면 "공존공간이 뭐지? 건물 이름?"이라는 혼란 발생 가능.

**대응:**
- 초기부터 "공존 = 플랫폼 브랜드" / "공존공간 = 1호 운영 건물"로 분리
- 마치 "아마존 = 플랫폼" / "아마존 북스 = 1호 매장"처럼

### 위험 5: 기존 SaaS와의 포지셔닝 충돌

**심각도: 중**

캐시노트, 보스몬, 얼마에요 등 이미 각 영역에 전문 플랫폼이 있다. "다 해줍니다"는 "다 어중간합니다"로 들릴 수 있다.

**대응:**
- 모든 기능을 직접 만들지 않고, 기존 서비스와 연동하는 **허브** 전략
- "공존은 이 모든 걸 하나로 연결합니다" (직접 제공 vs 연결 허브)
- 단, 핵심 기능(정산, 임대 관리)은 직접 보유

## 6. 강점 / 약점 / 개선제안 (종합)

### 강점 3가지

| # | 강점 | 설명 |
|---|------|------|
| 1 | **진짜 비어 있는 포지션** | 건물주-임차인 양면 플랫폼은 글로벌에도 없다. "발명"이 아니라 "발견"이라 더 강력하다 |
| 2 | **CAC 제로의 유통 구조** | 임대차계약이 곧 온보딩이다. 별도 마케팅 없이 임차인을 확보할 수 있는 구조 |
| 3 | **'공존'이라는 이름의 브랜드 자산** | 사업의 본질과 이름이 완벽하게 일치하는 경우는 드물다. 이 이름을 바꾸면 안 된다 |

### 약점 3가지

| # | 약점 | 설명 |
|---|------|------|
| 1 | **'건물주'라는 단어의 감정적 부채** | 한국 사회에서 건물주 = 갑. 이 인식을 뚫는 데 상당한 브랜드 에너지가 필요하다 |
| 2 | **임차인 데이터 불안** | "내 매출을 건물주가 본다"는 공포를 해소하지 못하면 온보딩 자체가 안 된다 |
| 3 | **"다 해준다"의 신뢰도 문제** | 급여, 세무, 장부, 마케팅까지 -- 소규모 플랫폼이 이걸 다 잘 할 수 있다는 걸 어떻게 믿게 하나 |

### 개선 제안 3가지

**제안 1: "건물주"를 브랜드 전면에서 빼라**

현재 메시지: "건물주가 임차인의 사업 성공을 돕는 모델"

개선 메시지: **"이 건물에 입주하면, 사업 운영이 쉬워집니다"**

주어를 "건물주"에서 "건물"로 바꾸는 것만으로 권력관계가 사라진다. 건물이 스마트해지는 거지, 건물주가 간섭하는 게 아니다. 마치 "이 아파트에 입주하면 헬스장이 무료다"처럼 자연스러운 건물의 부가서비스로 포지셔닝.

**제안 2: 임차인 온보딩 순서를 "무료 도구 먼저, 건물 연결 나중에"로 설계하라**

현재 흐름 (추정): 임대차계약 -> 플랫폼 가입 -> 도구 사용

개선 흐름: **무료 도구 사용(장부/급여) -> 가치 체감 -> "이 건물이면 프리미엄 기능도 무료" -> 건물과 연결**

임차인이 먼저 도구의 가치를 느끼고, 그 다음에 "아 이게 건물이랑 연결되면 더 좋아지는구나"를 깨닫게 하라. 순서가 바뀌면 "감시 도구"가 된다.

**제안 3: 론칭 콘텐츠를 "건물주 비전"이 아니라 "임차인 고충 해결"로 시작하라**

현재 방향 (추정): "로하의 비전은..." / "건물주와 임차인의 상생..."

개선 방향: **소상공인 생존 다큐멘터리 형식**

예시 콘텐츠:
> "새벽 1시, 장사 끝나고 엑셀 켜는 사장님. 알바비 계산, 카드 정산 확인, 부가세 자료 정리... 장사보다 서류가 더 힘들다는 말, 과장이 아닙니다. 그래서 만들었습니다."

이 콘텐츠에 "건물주"라는 단어는 한 번도 안 나온다. 먼저 공감시키고, 나중에 구조를 설명하라.

## 7. 미애의 최종 의견

이 확장, **방향은 맞다.** 시장에 진짜 빈자리가 있고, 공존이라는 이름이 그 자리에 딱 맞는다.

하지만 **말하는 순서**를 틀리면 죽는다.

```
[틀린 순서]  "건물주가 플랫폼을 만들었다" -> "임차인에게 도움이 된다"
[맞는 순서]  "소상공인이 이렇게 힘들다" -> "이걸 해결하는 도구가 있다" -> "알고 보니 건물 자체가 이 도구를 품고 있었다"
```

영웅은 건물주가 아니다. **영웅은 임차인(소상공인)**이다.
공존은 영웅의 무기를 만들어주는 대장장이다.
건물주는 그 무기를 무료로 제공하는 후원자다.

이 서사 구조를 지키면, "건물주가 감시한다"가 아니라 "이 건물은 사장님을 돕는다"가 된다.
