## 2. 경쟁 분석: 캐시노트/토스 vs 공존공간

### 2-1. 포지셔닝 비교

| 항목 | 캐시노트 | 토스플레이스 | 공존공간 |
|------|---------|-------------|---------|
| **사용자 수** | 210만+ 사장님 | 수십만 [추정] | 3개 임차인 (파일럿) |
| **진입점** | 카드매출 조회 (데이터) | POS (하드웨어+소프트웨어) | 월정산 자동화 (운영) |
| **핵심 가치** | "내 매출이 얼마인지 알려준다" | "주문-결제를 처리해준다" | "건물주-임차인 정산을 자동화한다" |
| **데이터 범위** | 매출/지출 조회 | POS 매출+주문 | 매출+지출+공과금+수수료+인건비+재고 통합 |
| **건물주 기능** | 없음 | 없음 | 핵심 타겟 |
| **임차인간 데이터 연결** | 없음 (각 사업장 독립) | 없음 | 있음 (동일 건물 내 통합) |
| **수익모델** | 광고+구독+컨설턴트 수수료 | 결제수수료+구독 | 미정 (파일럿 단계) |

### 2-2. 데이터 차별점 (핵심)

캐시노트와 토스가 **절대 가지지 못하는 데이터**:

| 데이터 | 캐시노트 | 토스 | 공존공간 |
|--------|---------|------|---------|
| 건물주의 고정비 구조 (대출이자, 관리비, 감가상각) | X | X | O |
| 임대료 vs 임차인 매출 비율 | X | X | O |
| 같은 건물 내 업종별 매출 비교 | X | X | O |
| 공과금의 임차인별 배분 | X | X | O |
| 건물 단위 총수익률 | X | X | O |
| 공실 발생 조기 신호 (매출 하락 추이) | X | X | O |

**결론: 캐시노트는 "소상공인 개인의 매출 데이터", 공존공간은 "건물 생태계의 운영 데이터"를 보유한다. 경쟁이 아니라 레이어가 다르다.**

## 3. 고객 가치: CAC/LTV 추정

### 3-1. 양면 네트워크 구조의 경제학

```
건물주 1명 확보 → 임차인 3~5명 자동 유입 (건물주가 요청/권유)
                → 건물주 CAC로 임차인 CAC를 0에 가깝게 만들 수 있음
```

| 지표 | 추정값 | 근거 |
|------|--------|------|
| 건물주 CAC | 50~100만원 [추정] | B2B SaaS 일반 CAC 참고. 초기는 직접 영업(고비용), 이후 레퍼럴 기대 |
| 임차인 CAC | 0~5만원 [추정] | 건물주가 도입하면 임차인은 따라옴. 추가 마케팅 불필요 |
| 건물주 LTV (보수적) | 120만원 [추정] | 월 2만원 x 60개월 (5년 유지) |
| 건물주 LTV (기본) | 360만원 [추정] | 월 3만원 건물주 + 임차인 3명 x 월 1만원 x 60개월 |
| 건물주 LTV (낙관적, 금융연계 포함) | 600만원+ [추정] | SaaS + 보험/대출 연계 수수료 포함 |
| LTV/CAC 비율 (기본) | 3.6~7.2x | 건강한 SaaS 기준 3x 이상이면 양호 |

### 3-2. 벤치마크 비교

| 서비스 | ARPU/년 | 고객 유지율 | 비고 |
|--------|---------|------------|------|

| 공존공간 (목표) | 36~72만원/건물 [추정] | 목표 80% (4개 서비스 제공 시) | Square 유지율 데이터 적용 |

### 3-3. 제품 수와 유지율의 관계

Square 데이터에 따르면:
- 1개 제품 사용: 이탈률 ~70%
- 4개 제품 사용: 이탈률 ~20% (유지율 80%)

공존공간이 제공 가능한 4개 서비스:
1. 월정산 자동화
2. 공과금 정산
3. 임차인 매출 리포트
4. 유지보수/운영 관리

**4개를 모두 쓰게 만들면 유지율 80%를 기대할 수 있다.**

## 4. 데이터 전략: 지금 쌓고 있는 데이터로 뭘 더 할 수 있는가

### 4-1. 현재 축적 중인 데이터

| 데이터 | 수집 방식 | 축적 기간 |
|--------|----------|----------|
| 미식가의주방 일별 매출 (카드/현금) | OKPOS 자동수집 | 수개월 |
| 포토인더박스 매출/지출 | 노션 DB 수동입력 | 수개월 |
| 재생전술 클래스/이벤트 매출 | 노션 DB 수동입력 | 수개월 |
| 공과금 (전기/가스/수도/통신) | 메일 자동수집 + 웹 크롤링 | 수개월 |
| 구매내역 (온라인/오프라인) | 노션 DB 수동입력 | 수개월 |
| 인건비 | config.js 고정값 | 수개월 |

### 4-2. 데이터 활용 단계 (숫자 기반 로드맵)

**Level 1: 현재 — 자동화 (가치: 시간 절감)**
- 월정산 자동 생성: 월 4~8시간 수작업 → 10분
- ROI: 인건비 기준 월 10~20만원 상당 절감 [추정]

**Level 2: 6개월 후 — 인사이트 (가치: 의사결정 지원)**
- 임차인 매출 트렌드 분석 (전월 대비, 전년 동월 대비)
- 공과금 이상치 탐지 (전월 대비 20% 이상 증가 시 알림)
- 건물 수익률 자동 계산 (총수입/총비용)
- 임차인별 수익 기여도 순위

**Level 3: 12개월 후 — 예측 (가치: 리스크 관리)**
- 12개월 데이터 축적 시 → 계절성 분석 가능
- 매출 하락 추세 감지 → 공실 위험 조기 경보
- 공과금 예측 → 정산 금액 사전 안내

**Level 4: 24개월 후 — 금융 연계 (가치: 새로운 수익원)**
- 매출 데이터 기반 임차인 신용 평가
- 매출보전보험 트리거 설계 (파라메트릭 보험: 글로벌 시장 $21B, 연 13% 성장 [출처: 매출보전보험 리서치])
- 매출 연동 임대료(Percentage Rent) 모델 데이터 기반 구현
- 보증금 대출/보험 연계

### 4-3. 데이터 자산 가치 추정

| 데이터 자산 | 활용처 | 수익 잠재력 |
|------------|--------|------------|
| 건물 수익률 벤치마크 | 부동산 투자자/금융기관 | 데이터 라이선스 수익 [추정] |
| 업종별 매출 트렌드 | 상권 분석/입지 선정 | KB 오픈업, 네이버 부동산에 없는 "건물 내부" 데이터 |
| 임차인 매출 vs 임대료 비율 | 적정 임대료 산정 | 건물주/임차인 양쪽에 가치 |
| 공과금 패턴 | 에너지 효율 컨설팅 | ESG 관련 수요 증가 중 |

## 5. 위험 조기 감지: 숫자상 위험 신호 5가지

### 위험 1: N=3 문제 (데이터 부족)

| 현실 | 위험 수준 |
|------|----------|
| 현재 임차인 3개, 건물 1동 | **높음** |
| 통계적 유의미한 패턴 도출에는 최소 30~50개 사업장 필요 [추정] | |
| "플랫폼"이라 부르려면 최소 10동 이상의 건물 데이터가 필요 | |

**조기 감지 지표**: 1년 내 건물 5동 미만이면 플랫폼 전략은 시기상조. 도구(tool) 전략으로 선회해야 한다.

### 위험 2: 건물주 획득 속도

| 시나리오 | 건물주 확보 수 (1년) | 판단 |
|----------|---------------------|------|
| 낙관 | 20동 | 플랫폼 가능성 확인 |
| 기본 | 5~10동 | 도구로서의 가치는 증명 가능 |
| 비관 | 3동 미만 | 제품-시장 적합성(PMF) 미달 |

소상공인 디지털 전환율 27.2% [출처: KDI]. 건물주의 디지털 수용도는 이보다 낮을 가능성이 높다 [추정]. 건물주 평균 연령이 50대 이상인 점을 감안하면, 온보딩 저항이 클 수 있다.

### 위험 3: 수익화 시점의 캐시번 (Cash Burn)

| 항목 | 추정 |
|------|------|
| 플랫폼 개발+운영 월 비용 | 300~500만원 [추정] (1~2인 개발 기준) |
| BEP 도달 필요 건물주 수 (월 3만원 ARPU) | 100~170동 |
| BEP 도달 예상 시점 | 18~36개월 [추정] |
| 그 전까지 필요 자금 | 5,400만~1.8억원 [추정] |

**조기 감지 지표**: 월간 반복매출(MRR)이 6개월 연속 성장하지 않으면 수익 모델 재검토가 필요하다.

### 위험 4: 기존 플레이어의 진입

| 위협 | 가능성 | 대응 시간 |
|------|--------|----------|
| 캐시노트가 건물주 기능 추가 | 중 | 6~12개월 [추정] |
| 토스가 임대관리 진출 | 중~저 | 12~24개월 [추정] |
| 네이버/카카오 부동산이 관리 기능 확장 | 저 | 24개월+ [추정] |
| 해외 프롭테크(Bidrento 등) 한국 진출 | 저 | 12~24개월 [추정] |

캐시노트의 210만 사용자 기반은 위협이지만, 그들의 데이터 구조가 "개별 사업장 단위"라는 점이 공존공간의 방어벽이다. "건물 단위 통합"으로 전환하려면 캐시노트도 상당한 아키텍처 변경이 필요하다 [추정].

### 위험 5: 공존 가치와 수익 모델의 충돌

| 시나리오 | 위험 |
|----------|------|
| 매출 데이터를 임차인 불리하게 활용 (임대료 인상 근거) | 임차인 이탈 + 브랜드 훼손 |
| 건물주 편향 기능 설계 (감시/통제 도구화) | 양면 플랫폼 붕괴 |
| 금융 연계 시 과도한 수수료 | 소상공인 72.6%가 이미 매출 감소 중 [출처: 소상공인연합회] |

**이 위험은 숫자 이전의 문제다. 공존 가치를 훼손하는 수익화는 영희가 분석을 거부한다.**

## 6. 강점 3개, 약점 3개, 개선 제안 3개

### 강점

| # | 내용 | 근거 |
|---|------|------|
| 1 | **글로벌에도 없는 양면 포지션** | 해외 사례 조사 결과, 건물주에게 임차인 사업 운영 도구를 제공하는 플랫폼은 발견되지 않음. Square는 소상공인만, HqO는 오피스 빌딩만 서빙한다. |

| 3 | **이미 작동하는 MVP** | monthly-report.js가 실제 3개 임차인의 정산을 처리하고 있음. 청사진 문서에 따르면 새 업체 적용에 4~6시간 소요. 이론이 아니라 동작하는 코드가 있다. |

### 약점

| # | 내용 | 근거 |
|---|------|------|
| 1 | **N=3의 데이터 한계** | 건물 1동, 임차인 3개로는 시장 검증이 불가. 통계적 유의성도 없고, 패턴 분석도 불가능하다. |
| 2 | **수익 모델 부재** | 현재 매출 0원. BEP까지 100~170동 필요. 캐시노트조차 210만 사용자에서 수익화에 고전 중인 시장이다. |
| 3 | **건물주 온보딩 검증 미달** | 소상공인 디지털 전환율 27.2%. 건물주(대체로 고연령)의 디지털 수용도는 이보다 낮을 가능성 높음. "기술 좋다"와 "쓸 사람이 있다"는 다른 문제다. |

### 개선 제안

| # | 제안 | 기대 효과 | 투입 비용 |
|---|------|----------|----------|
| 1 | **6개월 내 건물 5~10동 파일럿 확보** — 수원/경기 소규모 상가건물주 대상 무료 도입 후 데이터 축적. 숫자가 모이면 그때 플랫폼 판단. | PMF 검증 + 데이터 30개 사업장 이상 확보 | 영업 인력 1명 + 도입 지원 시간 (업체당 4~6시간) |
| 2 | **건물주 ARPU를 월 3만원에서 시작** — 정산 자동화(월 1만) + 공과금 관리(월 5천) + 매출 리포트(월 5천) + 유지보수 관리(월 1만). 4개 서비스 묶음으로 유지율 80% 타겟. | Square의 "4개 제품 = 80% 유지율" 전략 검증 | 기능 개발 2~3개월 |
| 3 | **캐시노트/토스와 경쟁이 아닌 연동 전략** — 임차인 매출 데이터는 캐시노트 API 또는 POS 연동으로 수집. 공존공간은 "건물 레이어" 에만 집중. 소상공인 도구를 처음부터 만들지 않는다. | 개발 비용 절감 + 임차인 저항 최소화 | API 연동 개발 1~2개월 |

## 7. 최종 판단 (숫자 요약)

| 질문 | 영희의 답 |
|------|----------|
| 시장은 있는가? | 있다. 건물주 100만+, 임차인 500만+. 시장 규모 최소 600억, 최대 4,800억원 [추정]. |
| 경쟁자와 다른가? | 다르다. "건물 단위 양면 데이터"는 캐시노트/토스가 구조적으로 갖지 못한다. |
| 지금 할 수 있는가? | 도구(tool)로는 지금 바로 가능. 플랫폼은 아직 이르다. |
| 돈이 되는가? | 아직 모른다. 검증이 필요하다. N=3으로는 판단 불가. |
| 위험은 무엇인가? | 건물주 온보딩 실패, 수익화 지연, 기존 플레이어 진입. 가장 큰 위험은 "검증 전에 확장하는 것"이다. |
