## 0. 결론 — Conditional-Go

법적으로 **전진 가능하다. 단, 다섯 가지 조건문을 동시에 충족해야 한다.**

> **"미식가 사장님과의 서면 합의 + 개인정보 처리 위탁 구조 + 전화번호 마스킹 + 외부 빅데이터의 '내부 분석 한정' 사용 + 임대차 관계와 자문 관계의 명시적 분리"** — 이 다섯이 동시에 갖춰지면 v0.1은 안전하다. 하나라도 빠지면 위험하다.

특히 v0.1은 "1개 매장, 무상 실험"이라는 좁은 사실관계 덕분에 합법 영역에 머무를 수 있다. **v0.2로 확장하는 순간 거의 모든 항목의 위험도가 한 단계씩 올라간다는 점을 미리 못 박아둔다.**

## 2. 리스크 항목별 분석

### 2-1. 외부 빅데이터의 사용 경계 (KT 유동인구, 신한카드)

**핵심 판정:** v0.1처럼 **"미식가 1곳을 위한 내부 분석 자료"로만 쓰면 합법.** "행궁동 상권 리포트를 다른 사장님에게 판매"하는 순간 위법 가능성이 급격히 올라간다.

| 데이터 | 표준 약관의 일반 구조 | v0.1에서 안전한 사용 | 위법 가능성 |
|---|---|---|---|
| KT 빅데이터 (LOCA 등) | 가공·재판매·제3자 제공 금지. 보고서 형태로 한정된 인용 허용 | "미식가 사장님 미팅 자료" 1회성 인용 | 다른 매장에 동일 보고서를 재활용·복제·판매하면 위반 |
| 신한카드 빅데이터 (My샵·신한SOL 등) | 동일. 분석 결과의 외부 공표 시 신한카드 표기·승인 요구하는 경우 다수 | 사장님 1명에게 구두 설명 + A4 1장 인용 | 동일. 캡처/2차 가공물 외부 공표 시 사전 승인 필요할 수 있음 |

**필수 조치:**
1. 이승훈이 실제 사용 계정의 **이용약관 원문을 캡처·보관**한다 (분쟁 시 1차 증거).
2. v0.1 리포트 풋노트에 **"본 자료는 미식가의주방 내부 검토용이며 무단 복제·배포를 금지함"** 명기.
3. v0.2에서 "다른 사장님에게 판매" 단계로 갈 때는 **반드시 통신사·카드사와 별도의 데이터 활용 계약(또는 API 라이선스)을 다시 협상**한다. 이때 4/6에 언급한 데이터바우처 사업이 비용 커버 수단이 된다.

### 2-2. 미식가 내부 데이터 취급 (OKPOS, 네이버 예약)

**핵심 판정:** **이승훈이 "개인 자격"으로 데이터를 다루면 위험. "공존공간 또는 슈파라는 사업체가 미식가로부터 처리 위탁받은 수탁자"로 다루면 안전.**

쟁점은 두 가지다.

**(a) 처리자 지위의 충돌**

미식가 사장님은 이미 "개인정보 처리자"다 (네이버 예약을 통해 손님 전화번호를 수집·보유). 이승훈이 이 데이터를 받아 분석한다면, 개인정보보호법상 다음 중 하나여야 한다.

| 구조 | 법적 성격 | 필요 문서 |
|---|---|---|
| **(권장) 처리 위탁** | 미식가가 위탁자, 슈파(이승훈)가 수탁자. 정보주체(손님)에게는 위탁 사실을 미식가가 공개 (개인정보보호법 제26조) | **개인정보 처리위탁 계약서 1장** + 미식가의 개인정보처리방침에 위탁 사실 명기 |
| 제3자 제공 | 손님 한 명 한 명의 별도 동의 필요 | 비현실적 (수백 명 재동의) |
| 가명처리 후 전달 | 개인정보보호법 제28조의2. 통계·연구 목적 가능 | 전화번호 해시화/마스킹 후 전달 |

**v0.1에서 가장 현실적인 조합:** "처리위탁 계약서 + 전화번호 가명처리"의 이중 안전장치.

**(b) 수집 항목의 최소화**

v0.1 분석에 진짜 필요한 것은 "예약 시간, 인원수, 노쇼 여부, 매출 짝짓기"다. **전화번호 원문은 필요 없다.** 매칭 키로 쓰려면 해시값(SHA-256)으로 변환하면 충분하다.

**필수 조치:**
1. **개인정보 처리위탁 계약서 1장** 체결 (A4 2~3장 분량으로 충분, 표준 양식 존재)
2. **전화번호 마스킹 파이프라인**: 네이버 예약에서 추출한 데이터의 전화번호 칼럼은 추출 직후 해시 또는 마스킹(010-****-1234) 처리하고 원문은 즉시 삭제
3. **정보주체 고지**: 미식가 사장님의 매장 내 고지문 또는 네이버 예약 안내문에 "예약 데이터는 매장 운영 분석을 위해 공존공간(주)에 처리 위탁될 수 있습니다" 한 줄 추가

### 2-3. LLM API 국외 이전 리스크

**핵심 판정:** 개인정보보호법 제28조의8(개인정보 국외 이전)은 **정보주체 동의 또는 처리위탁 계약 내 국외 이전 명시**를 요구한다. **마스킹된 데이터(가명정보)만 LLM에 보내면 이 조항의 적용에서 사실상 벗어난다.**

| 시나리오 | 합법성 |
|---|---|
| 전화번호 원문을 그대로 OpenAI/Anthropic API에 송신 | **위법 가능성 높음.** 국외 이전 동의·고지 부족 + 처리 목적 외 이용 |
| 전화번호를 해시·마스킹한 후 API 송신 | 가명정보로 분류되어 보다 자유로운 활용 가능 |
| 매출·예약 통계만 (고객 식별 정보 제외) API 송신 | 사실상 안전 |

**필수 조치:**
1. **LLM 입력 전 자동 PII(개인식별정보) 스크러빙**: 전화번호, 이름, 카드번호 패턴을 정규식으로 잡아내고 마스킹한 후에만 프롬프트에 삽입.
2. 처리위탁 계약서에 **"수탁자는 분석 목적으로 가명처리된 데이터를 해외 클라우드 AI 서비스에 전송할 수 있다"** 한 줄을 명시. (사장님이 알고, 동의한 사실의 증거)

### 2-4. 네이버 예약 데이터 접근 방법 — 가장 까다로운 지점

**핵심 판정:** 공식 API가 없을 가능성이 높다. **그렇다면 "사장님 본인의 화면에서 사장님이 직접 다운받은 CSV/엑셀을 슈파에 제공하는 구조"가 유일한 안전 경로다.** 이승훈이 사장님 ID/PW로 로그인하거나 자동 스크래핑을 돌리는 순간 세 가지 법이 동시에 걸린다.

| 행위 | 법적 리스크 |
|---|---|
| 사장님 계정으로 이승훈이 로그인 | **정보통신망법 제48조(정보통신망 침해)** — "정당한 접근권한 없이"의 해석. 사장님이 동의해도 약관 위반이면 권한이 부정될 여지 있음. **네이버 약관상 계정 양도/공유는 거의 모든 서비스에서 명시적 금지** |
| 자동 스크래핑 봇 운영 | 네이버 약관 위반 + 정보통신망법 위반 가능성 + 컴퓨터프로그램보호법 쟁점 |
| **사장님이 직접 화면에서 다운로드 → 슈파에 파일 전달** | **합법.** 사장님이 자기 데이터를 자기 의지로 내보내는 행위는 네이버가 막을 수 없음 |

**필수 조치:**
1. v0.1 단계에서는 **"월 1회 미팅 시 사장님이 직접 다운받은 엑셀을 USB/메일로 받는다"**는 수동 방식으로 시작. 자동화의 유혹을 참는다.
2. v0.2 자동화가 필요하면 네이버에 공식 API 문의를 먼저 넣고, **거절 회신을 문서로 보관**한 뒤에 다른 경로를 검토한다.
3. **사장님 계정으로 직접 로그인은 절대 하지 않는다.** 한 번이라도 시작하면 모든 v0.1 결과물이 "위법한 정보 수집의 산물"이 되어 증거능력을 잃는다.

### 2-5. 건물주-임차인 관계의 충돌 — 이번 v0.1의 진짜 사각지대

**핵심 판정:** **이것이 가장 미묘하고 가장 중요한 쟁점이다.** 이승훈은 동시에 두 모자를 쓴다 — (i) 미식가가 입주한 건물의 임대인, (ii) 미식가의 매출 데이터를 분석하는 자문 제공자. 이 두 역할이 분리되지 않으면 다음 두 가지 사고가 일어날 수 있다.

| 시나리오 | 법적 결과 |
|---|---|
| 사장님이 1년 후 매출 데이터가 임대료 협상·갱신거절·임대료 인상의 근거로 쓰였다고 주장 | 상가임대차보호법상 갱신거절·차임증감청구 분쟁. 데이터 분석 자체가 "우월적 지위 남용"으로 채색될 수 있음 |
| 사장님이 슈파 자문 결과대로 했는데 매출이 떨어졌다고 주장 | 자문 책임 분쟁. "무상 실험"임을 입증할 수 없으면 손해배상 소지 |

**필수 조치 — 두 관계의 명시적 분리 (1장짜리 합의서):**

```
1. 본 합의서는 "미식가의주방 데이터 분석 실험"에 관한 것이며,
   임대차 계약과 완전히 별개의 관계입니다.
2. 본 분석 결과는 임대료 산정·갱신·증액 협상에
   어떤 형태로도 사용되지 않습니다.
3. 본 분석은 무상 실험이며, 사장님은 언제든 중단할 수 있고
   슈파는 결과에 대한 어떠한 보증·책임도 지지 않습니다.
4. 사장님은 분석 데이터의 삭제를 언제든 요청할 수 있습니다.
5. 분석 데이터는 사장님 매장의 전략 수립 외 목적으로
   사용되지 않으며, 제3자에게 양도되지 않습니다.
```

이 한 장이 v0.1의 모든 미시 리스크를 막는 **가장 저렴하고 가장 효과적인 방패**다. 미팅 전 사장님과 함께 읽고 사인 받는 데 10분이면 족하다.

## 4. 정책 활용 기회

v0.1은 무상 실험이지만, **"법적 안전망 구축 비용"과 "데이터 사용료"는 정책자금으로 커버 가능하다.** 4/6에서도 일부 언급했지만, v5 프레이밍에 맞게 다시 골라낸다.

| 사업 | 주관 | v0.1~v0.2에서의 활용 가능성 | 신청 현실성 |
|---|---|---|---|
| **데이터바우처 (구매형/일반)** | 과기부·한국데이터산업진흥원 | KT/신한카드 빅데이터 구매 비용 최대 4,500만원 지원. v0.1에서는 무료 트라이얼로 충분하지만, **v0.2 확장 시 1순위로 신청** | 매년 봄 공모. 슈파를 "수요기업"으로 등록 |
| **AI 바우처 사업** | 과기부·정보통신산업진흥원 | LLM·AI 솔루션 도입 비용 최대 3억원 지원. 슈파의 AI 전략 레이어 개발 자금으로 활용 | 슈파가 법인화되어야 수요기업 자격 가능. **법인화 시점을 정책 신청 일정에 맞춰 설계** |

## 5. 강점 / 약점 / 개선 제안

### 강점 3개

1. **거시 규제가 사라졌다.** 보험업법·자격법(세무사·노무사)·매출연동 임대료 같은 고난도 규제가 v5에서 모두 빠지면서, 슈파는 "AI가 전략을 제안한다"는 단일 행위만 남았다. 이 단일 행위는 한국법상 무자격 침해를 일으키지 않는다 — "경영 컨설팅"은 아무 자격도 요구하지 않는 자유업이기 때문.
2. **사실관계가 좁다.** 1개 매장, 무상, 월 1회 미팅, 이승훈 직접 진행. 이 좁은 사실관계 안에서는 거의 모든 데이터 리스크가 "처리위탁 + 가명처리"라는 두 장치로 막힌다.
3. **정책 친화적 스토리.** "AI가 소상공인의 전략 참모가 된다"는 v5 메시지는 중기부·과기부·소진공의 현행 모든 디지털 전환 정책 키워드와 정확히 부합한다. 공모 사업 평가서에 그대로 옮겨 적을 수 있는 문장이다.

### 약점 3개

1. **v0.1과 v0.2 사이에 법적 단층선이 있다.** 1곳에서 합법인 것이 10곳으로 확장되는 순간 위법으로 바뀌는 항목들이 있다. 외부 빅데이터 재배포, LLM 국외 이전, 네이버 데이터 자동 수집 모두 그렇다. **v0.1을 성공시키면 자동으로 v0.2가 되는 게 아니라, v0.2를 위해 별도의 법적 재설계가 필요하다.**
2. **이승훈이 "개인" 자격으로 데이터를 다룬다는 사실 자체.** 슈파가 아직 법인이 아니고 공존공간(주)도 부동산법인이다. 어느 이름으로 데이터 처리위탁을 받는지가 모호한 채로 시작하면 사고 시 책임 주체가 불분명해진다.
3. **건물주라는 정체성이 자문 관계를 오염시킬 수 있다.** 사장님이 1년 후 무엇을 주장할지 예측 불가능하다. 4/6 시점에는 이 충돌이 보이지 않았는데, v5에서 "사장님과 직접 미팅하며 자문"이라는 형태가 명확해지면서 표면화됐다.

### 개선 제안 3개

1. **첫 미팅 전에 1장짜리 합의서 + 처리위탁 계약서를 받아라.** 둘 다 표준 양식이 있고 합쳐도 A4 4장을 넘지 않는다. 사장님과 함께 30분 안에 끝낼 수 있다. 이게 v0.1의 모든 미시 리스크를 막는 가장 싼 방패다.
2. **"전화번호 원문은 슈파 시스템에 절대 들어오지 않는다"를 v0.1의 기술적 헌법으로 못 박아라.** 추출 단계에서 즉시 해시화. 이 한 줄이 LLM 국외이전·개인정보 유출·네이버 약관 충돌 세 가지를 동시에 줄인다.
3. **이름의 명확화 + 정책 무료자원 즉시 활용.** 슈파가 어느 법인의 사업인지(공존공간 산하 실험인지, 별도 법인 예정인지)를 v0.1 종료 전에 확정한다. 동시에 개인정보보호위 무료 컨설팅과 중기부 비즈니스지원단 무료 법률자문을 즉시 신청해 처리위탁 계약서·합의서를 검토받는다. **법무비 0원, 시간 2주, 안전망 95% 확보.**

## 주희의 최종 의견

법은 장애물이 아니라, 가장 창의적인 길을 찾기 위한 지도다.

v5/v0.1로 좁아진 슈파는, 4/6에 봤던 거대한 규제 그림자에서 벗어났다. 대신 "1장짜리 합의서 + 1줄짜리 마스킹 코드 + 1번의 무료 법률 자문"이라는 매우 작은 단위의 안전장치들이 새로 필요하다. 이것들은 비싸지 않고, 빠르고, 사장님과의 신뢰를 오히려 강화한다.

전진해도 좋다. 단, 아래 다섯 줄을 v0.1 시작 조건으로 못 박는다.

> 1. 미식가 사장님과 **1장짜리 관계분리 합의서** 서명
> 2. **개인정보 처리위탁 계약서** 체결
> 3. **전화번호 즉시 해시화** 파이프라인
> 4. 외부 빅데이터는 **"미식가 1곳 내부 검토용"** 한정 사용
> 5. 네이버 예약 데이터는 **사장님이 직접 다운로드한 파일**만 수령

이 다섯이 갖춰지면 v0.1은 법정에서도 흔들리지 않는다. 갖춰지지 않으면, MVP의 검증 결과가 무엇이든 "위법한 실험의 산물"이라는 꼬리표가 따라붙는다. 이 꼬리표는 v0.2 정책 자금 신청부터 향후 투자 유치까지 전 과정에서 발목을 잡는다.

**판정: Conditional-Go. 다섯 조건을 시작 전 충족하라.**
