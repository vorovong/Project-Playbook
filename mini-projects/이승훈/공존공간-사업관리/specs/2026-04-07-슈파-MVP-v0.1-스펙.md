> 작성일: 2026-04-07

> 목적: 슈파의 첫 버전(v0.1)이 무엇이고, 무엇이 아닌지, 어떻게 검증할지 명확히 함

---

## 0. 한 줄 요약

> **미식가의주방 하나에서, KT 유동인구 + 신한카드 + OKPOS + 네이버 예약을 합쳐 AI가 "이 타겟을 노리세요 + 왜 + 이번 달 액션 3가지"를 제안하는 v0.1을 만든다. 승훈이 직접 사장님과 미팅하며 제안 언어를 검증하고, 사장님이 한 가지 액션을 실제로 실행할 때까지를 성공으로 본다.**

---

## 1. 슈파가 풀 문제 (재확인)

> **"소상공인은 데이터도 있고 소스도 알려져 있다. 그런데 '그래서 우리 가게는 누구에게 뭘 팔아야 하는가?'라는 전략적 답을 내주는 사람이 없다."**

**슈파의 포지션:** 운영 도구(Square/Toast)도, 대시보드(캐시노트)도 아닌 **"AI 전략 참모"**

**v0.1에서 검증할 핵심 가설:**
AI가 외부 빅데이터 + 내부 데이터를 결합해서 낸 **"타겟 + 근거 + 액션" 제안**을, 실제 사장님이 **읽고, 이해하고, 한 가지라도 실행하는가?**

→ 이게 되면 v0.2 확장 가치 있음. 안 되면 데이터 아무리 늘려도 소용없음.

---

## 2. 결정된 범위 (6가지)

### 2-1. 접근 방식 — (나) MVP 작게, 빠른 검증
- **이유:** 진짜 검증 대상은 "AI 전략 레이어의 전달 방식"이지 데이터 양이 아님. 데이터 5개 전부 연결하느라 2~3개월 쓰면 검증이 지연되고, 최악의 경우 "리포트 자체가 안 읽히는" 진짜 원인을 너무 늦게 발견함.
- **원칙:** 데이터는 나중에 쉽게 추가 가능. AI 제안 방식·언어는 여러 번 고쳐야 맞는 걸 찾음 — 이게 진짜 시간 드는 부분.

### 2-2. 외부 데이터 — 2개
- **KT 유동인구** — 행궁동에 누가(연령/성별/거주지) 몇 시에 있는가
- **신한카드 빅데이터** — 행궁동 음식점 결제자의 객단가·연령·소비 패턴
- **조합의 의미:** "누가 오는가 × 얼마 쓰는가" = 인구-소비 정합성 분석
- **보류한 3개:** 소진공 상권정보시스템, 오픈업(KB), 네이버 데이터랩 → v0.2에서 추가

### 2-3. 내부 데이터 — 2개 (둘 다)
- **OKPOS 매출** — 요일/시간대별 매출, 메뉴별, 객단가
- **네이버 예약** — 예약자 전화번호, 인원수, 예약 시간, 노쇼
- **이유:** 조합 C(인구×소비)가 성립하려면 내부에서도 **사람×매출** 짝이 필요. OKPOS만이면 "누구한테 벌었나" 없음, 네이버 예약만이면 "얼마 썼나" 없음.
- **매칭 로직:** 날짜 + 시간 + 인원수로 OKPOS 주문과 네이버 예약을 느슨하게 붙임. 완벽 매칭 불필요 (비예약 손님은 빠져도 OK).

### 2-4. 결과물 전달 방식 — (다) 카톡 + (마) 직접 미팅
- **1단계 (초기 3~4회):** 승훈이 (마) 직접 미팅에서 AI 리포트를 들고 가 사장님에게 설명. 얼굴 보고 반응·이해도·질문 수집. 제안 언어를 잡는 단계.
- **2단계 (언어 확정 후):** (다) 카카오톡 주간 메시지로 자동화.
- **대시보드 (나) 는 보류** — v0.2 이후.
- **이유:** 자동 메시지부터 보내면 "안 읽었어요"로 끝나서 진짜 원인을 모름. 사람이 직접 가서 "이거 이해되세요?" 해야 제안 언어의 문제가 잡힘.

### 2-5. AI 추천의 깊이 — 깊이 2 (결론 + 근거 + 액션 1~3개)

**표준 리포트 구조:**
```
[결론] 한 문장
미식가는 ___ 타겟을 중심으로 잡아야 합니다.

[근거] 3~4줄
- 행궁동 유동인구 상위: ___ (KT)
- 행궁동 음식점 객단가: ___ (신한카드)
- 미식가 현재 손님: ___ (내부)
- 갭: ___

[이번 달 액션 3가지]
1. ___
2. ___
3. ___
```

- **분량:** A4 한 장 이내. 15분 안에 설명 가능해야 함.
- **금지:** 깊이 1(한 줄만) → 행동 연결 없음. 깊이 3(풀 리포트) → 안 읽힘.

### 2-6. 첫 사용자 — (다) 미식가 먼저 → 대봉 2차
- **v0.1 기간:** 미식가의주방 1곳에만 집중
- **대봉통닭:** v0.1 검증 완료(L3 도달) 후 2차 투입 — "이거 미식가만의 얘기 아니네"를 증명하는 외부 검증 역할
- **사장님 미팅:** 월 1회, 이승훈이 직접 진행 (응해주신다는 것 확인 완료)

## 3. 성공 기준 (MVP 합격선)

### 3단계 성공 레벨

| 단계 | 기준 | 측정 방법 |
|---|---|---|
| **L1 — 이해** | 사장님이 리포트를 듣고 "이해됐어"라고 말함 | (마) 미팅에서 표정·질문·반응 관찰 |
| **L2 — 동의** | 제안 3가지 중 최소 1가지에 "해볼게"라고 함 | 미팅 말미 액션 합의 |
| **L3 — 실행** | 실제로 그 액션 하나가 실행됨 | 다음 미팅에서 확인 |

### MVP 합격선
- **최소 합격:** L3 **1회 이상 도달** = 한 번이라도 "AI 제안 → 실제 실행"이 성사되면 MVP는 성공.
- **이상적 합격:** L3이 2~3회 반복 → 제안 언어·구조가 안정적이라는 신호 → v0.2 데이터 확장 시작.
- **실패 신호:**
  - L1에서 막힘 ("무슨 말인지 모르겠어") → AI 제안 언어가 문제. 프롬프트·용어 재설계 필요.
  - L1~L2 통과하나 L3 못 감 ("좋긴 한데 못 하겠네") → 액션이 너무 무거움. 액션 크기 축소 필요.

### 의도적으로 제외한 것
- **매출 효과 측정** — 변수가 너무 많고 시간이 오래 걸림. v0.2 이후에 측정.
- **다매장 비교** — 대봉은 L3 도달 후 투입.
- **자동화 완성도** — 승훈이 수동으로 데이터 모으고 AI 돌려도 OK. 자동화는 L3 이후.

## 4. 다음 단계 (v0.1 구현 계획 초안)

> 실제 구현 계획은 별도 planning 문서로 분리. 여기는 큰 덩어리만.

### 빌드 순서 (의존성 기반)

1. **데이터 수집 파이프라인**
   - KT 유동인구 — 행궁동 반경, 연령/성별/시간대/거주지
   - 신한카드 빅데이터 — 행궁동 음식점 결제 프로필
   - OKPOS — 기존 수집 파이프라인(`okpos-sales.js`) 재활용
   - 네이버 예약 — 데이터 추출 방법 확인 필요 (API? 스크래핑?)

2. **매칭 로직**
   - OKPOS 주문 × 네이버 예약 = 날짜+시간+인원 기준 느슨한 조인

3. **AI 전략 레이어 (핵심)**
   - 4개 데이터를 받아 "깊이 2 리포트"를 생성하는 프롬프트 설계
   - 프롬프트는 대화 초기엔 수동 조정 전제 (승훈이 직접 튜닝)

4. **리포트 출력**
   - A4 한 장 PDF or Word → 승훈이 들고 갈 수 있는 물리적 형태

5. **미팅 진행 + 피드백 수집**
   - 첫 미팅 전에 승훈이 먼저 리포트 자가 검증 ("내가 봐도 이해되는가?")

### 오픈 이슈 (다음에 정할 것)
- 네이버 예약 데이터 접근 방법 (공식 API 유무 확인)
- KT/신한카드 데이터의 실제 접근 방식 — 승훈님이 써본 방식 그대로 가는지
- 첫 리포트 생성 목표 시점
- 사장님 첫 미팅 목표 시점

| 결정 | 값 | 이유 |
|---|---|---|
| 접근 방식 | (나) 작게, 빠른 검증 | AI 레이어 검증이 데이터 양보다 먼저 |
| 외부 데이터 | 조합 C: KT + 신한카드 | 인구 × 소비 정합성 |
| 내부 데이터 | OKPOS + 네이버 예약 (둘 다) | 사람 × 매출 짝 필요 |
