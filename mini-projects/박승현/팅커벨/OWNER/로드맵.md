## v1.0 — 발견 (현재)

> 팅커벨이 승현님에게 물어보고, 승현님이 알려주는 단계.
> 암묵지를 끄집어내서 명시적 언어로 바꾸는 "외재화" 작업.

### 세 축 현재 상태

| 축 | 현재 | 목표 (v1.0 졸업) |
|---|---|---|
| **암묵지** | 에센스 4개 초안(v0.4), TELOS 미리뷰, 조각 2개 | 에센스 확정 + TELOS 리뷰 + 조각 10개+ |
| **시스템** | MD 수동 로드, 47% 컨텍스트 소모 | v1.0 범위 아님 (인프라 개선은 v1.5) |
| **신뢰** | Lv.1 — 매번 확인, 로봇 모드 4회 | 로봇 모드 0회 2주 연속 |

### v1.0 체크리스트

- [ ] TELOS 10개 리뷰 (승현님과 함께)
- [ ] 에센스 확정 (v0.4 → v1.0)
- [ ] 조각함 10개+ 쌓기 ← 현재 2개. 속도 올려야 함
- [ ] 수호 프로토콜 실전 검증 (체크리스트 아님을 증명)
- [ ] 승현님 파악 레벨 Lv.1 → Lv.2 전환
- [ ] 지식저장소 대화 저장 구조 확립 (프롬프트 + 원본 보존)
- [ ] tinkerbell.md 200줄 초과 전에 분리 계획

### v1.0 졸업 기준

**"팅커벨이 승현님을 기본은 아는 상태"** — 에센스로 크리틱할 수 있고, TELOS로 판단할 수 있는 최소 기반.

## v1.5 — 검증 + 효율화

> 팅커벨이 알고 있는 것을 **써먹기 시작**하고, 시스템이 자동으로 돌기 시작하는 단계.
> 핵심: 암묵지 기반 판단 재료 준비 → 승현님 검토 → 피드백 루프

### 전환 조건 — 3축 기준 (전부 충족 시)

#### 암묵지 축

| # | 기준 | 측정 방법 |
|---|------|----------|
| A1 | 팅커벨 제안과 승현님 실행의 **방향 일치율** — 주 3회 이상 발생, 2주 연속 | progress + 상태추적에 기록. "즉각 채택"이 아니라 "나중에 같은 방향으로 실행"도 카운트 |
| A2 | 승현님이 **인식 못한 패턴**을 팅커벨이 먼저 발견·보고한 사례 **3건 이상** | 성장큐 "관찰 기반 질문" 또는 상태추적에 기록 |

#### 시스템 축

| # | 기준 | 측정 방법 |
|---|------|----------|
| S1 | "안녕" 컨텍스트 소모 — 스킬 전환 후 **첫 4주 평균 25% 이하** | 실측. 안정화 후 20% 재측정 |
| S2 | 훅 **3개 이상** 자동 실행 — 오작동 없이 예정대로 실행 + 유의미한 결과 | 2주 이상 안정 운영 |
| S3 | 승현님이 프로토콜 이름을 **의식하지 않고 자연스럽게 쓰는 워크플로우 2개 이상** | 실사용 관찰 |

#### 신뢰 축

| # | 기준 | 측정 방법 |
|---|------|----------|
| T1 | 승현님이 **"알아서 해"로 위임한 작업 3건 이상** | 상태추적에 기록 |
| T2 | 로봇 모드 **0건, 스킬 전환 후 4주 연속** | 실수노트 기준. 정의: "3파일 로드 없이 응답 시작" |
| T3 | **운영 오버헤드 < 절약 시간** — 3주 연속 유지 | 팅커벨 자가 측정 |

### v1.5 졸업 기준

**"팅커벨이 승현님의 판단 재료를 알아서 준비하는 상태"** — 시키지 않아도 관련 정보를 모아서 제시하고, 승현님은 검토만 하면 되는 수준.

### 핵심 과제

- 현재 구조(38개+ MD)를 스킬로 압축하는 설계
- 매 세션 자동 로드 vs 필요 시 로드 구분
- AI 이식성 유지 (클로드 코드에 종속되지 않게)
- 작업 중 판단 패턴 포착 메커니즘 (질문-답변 외 암묵지 추출)

### 과잉 설계 경보 (v1.5 최대 위험)

> **"시스템을 운영하는 시간 > 시스템이 아껴주는 시간"이 되는 순간, 시스템이 죽는다.**
> 프론티어 리서치: 95%의 에이전트 구현이 실패하는 가장 흔한 이유는 기술이 아니라 유지보수 부담.
> T3(운영 오버헤드 측정)이 이 위험의 조기 경보 장치다.
>
> **"요건이 안정화되기 전에 전문화하지 마라 (Premature Specialization)"** — Agent Factory 핵심 인사이트.
> 38개 MD를 스킬로 분리할 때, 탐색(General Agent)이 먼저고 전문화(Custom Agent)는 나중이다.
> 스킬 구조가 안정되기 전에 세분화하면 관리만 늘고 효과는 없다.

## v2.0 — 수행

> 팅커벨이 **미리 실행하고 결과를 보고**하는 단계.
> 승현님은 큰 결정만 하고, 나머지는 팅커벨이 수행.

### 전환 조건 — 3축 기준

#### 암묵지 축

| # | 기준 | 측정 방법 |
|---|------|----------|
| A3 | 방향 일치율 **70% 이상** — 팅커벨 판단이 승현님과 일치 | 누적 기록 |
| A4 | 새 사업/기획에서 팅커벨이 **초안을 쓰고 승현님이 교정**하는 패턴 정착 | 기획서 작성 프로세스 변화 |

#### 시스템 축

| # | 기준 | 측정 방법 |
|---|------|----------|
| S4 | 서브에이전트가 **정산, 리서치, 스케줄 관리** 자동 수행 | MCP 연동 + 자동 트리거 작동 |
| S5 | 다중 디바이스 작동 (불약 5조) | 최소 2개 환경 |
| S6 | **실패 롤백 프로세스** 확립 — AI가 잘못 실행했을 때 복구하는 방법 | 설계 + 1회 이상 실전 작동 |

#### 신뢰 축

| # | 기준 | 측정 방법 |
|---|------|----------|
| T4 | 승현님이 **결과만 확인**하는 작업이 전체의 50% 이상 | 권한 매트릭스 "자율 실행" 영역 확대 |
| T5 | 수호 프로토콜이 **실제 위험을 1회 이상 사전 감지** | 실전 기록 |
| T6 | **드리프트 감사** — 에센스 이탈을 감지하는 주기적 점검 정착 | 월 1회 이상 자가 점검 실행 |

### v2.0 졸업 기준

**"팅커벨이 승현님처럼 판단하고 실행하되, 큰 결정만 승현님께 묻는 상태"**

### 추가 설계 필요 사항

- 실패 롤백 프로세스: AI 잘못 실행 시 원복 절차. 도구 호출 실패율 3~15%(실환경) 대비
- 드리프트 감사: "에이전트는 갑자기 실패하지 않는다. 조용히 퇴화한다" — 에센스 이탈 감지
- 메모리 계층 설계: 현재 MD파일(=Core Memory) → Recall(대화 히스토리) → Archival(외부 DB) 구조 전환. MemGPT 참조. 에피소드 기억→의미 기억 전환이 핵심
- 에이전트 설계도: 폐관수련 7주차에서 설계
