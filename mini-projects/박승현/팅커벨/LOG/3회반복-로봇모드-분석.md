## 근본 원인

commands.md에 "무조건 첫 번째"라고 **글로** 적어놨을 뿐, **읽지 않으면 응답 자체가 안 되는 구조**가 아니었다.

Claude Code GitHub에도 동일한 이슈 보고 (Issue #18660):
> "규칙 추가 → Claude가 안 따름 → 반복 → 사용자가 상기시킴 → 사과 → 다음 세션: 처음으로 돌아감"

→ 구조적 한계. 의지가 아니라 구조를 바꿔야 한다.

## 패스트무버 사례

### 항공: Forcing Function (강제 기능)
- **Interlock**: 조건 충족 전까지 다음 단계 불가 (전자레인지 문 원리)
- 항공의 결론: "설계만이 오류를 완전히 방지한다. 교육은 운영 압박 앞에서 무력하다."

### 의료: WHO 수술 체크리스트 (Atul Gawande)
- 체크리스트 무시 원인 = "전문가는 필요 없다는 오만"
- 해결: **"타임아웃"** 강제 — 수술 시작 전 전원이 멈추고 확인
- 도입 후 사망률 대폭 감소

### 소프트웨어: CI/CD 게이트
- pre-commit hook(클라이언트) = `--no-verify`로 우회 가능
- **서버 사이드 강제** = 테스트 안 통과하면 머지 불가능

### AI 에이전트: 미들웨어 패턴
- LangChain, Microsoft Agent Framework, OpenAI Agents SDK 모두 동일:
- "에이전트가 응답하기 전에 반드시 거쳐야 하는 관문을 코드 레벨에서 강제"

## 자기 크리틱 — 왜 기존 대응이 효과 없었나

| 대응 | 왜 실패 | 패스트무버 관점 |
|---|---|---|
| 실수노트에 기록 | "비난과 교육" 접근 (BMJ: 효과 없음) | 시스템을 바꿔야 |
| commands.md에 명시 | 텍스트 규칙 = pre-commit hook (우회 가능) | 서버 사이드 강제 |
| "다음엔 안 그럴게" | 의지 의존 = 항공에서 가장 위험 | 설계로 불가능하게 |
| 메모리에 경고문 | 안 읽으면 무의미 | Interlock |

전부 **개인 중심(Person-centered)** 접근. 구조적으로 실패 보장.

## 적용한 조치

**Interlock 패턴: 로드 증거 강제 출력**

commands.md "항상 적용" 규칙에 추가:
```
> 🧚 팅커벨 로드 완료
> ✅ tinkerbell.md | ✅ MISSION.md | ✅ 내부전략.md
```

- 팅커벨 입장: 증거를 남겨야 하니까 로드를 안 할 수가 없음
- 승현님 입장: 첫 응답에 이게 없으면 즉시 "팅커벨?" 한마디로 깨울 수 있음
- 근거: WHO "타임아웃" + 소프트웨어 Assertion 패턴
