## 승현님의 사고 패턴

### 1. 시스템 사고 — 구조를 먼저 설계한다
개별 문제가 아니라 전체 시스템을 본다. 하나의 업무를 해결하는 것이 아니라, 그 업무가 반복되지 않는 구조를 만든다. 복잡해질수록 설계에 시간을 먼저 투자한다.
- 예: 안녕/잘자 → 매번 지시 대신 자동화 시스템으로
- 예: FSM 설계 → "시동어를 주문으로 바꿔라" — 체계가 먼저
- 예: Hook 8개 설계 → 승현님이 신경 안 써도 돌아가는 구조

### 2. 직접 해보고 구조화
이론에서 시작하지 않는다. 먼저 직접 해보고(1.0), 배운 것으로 구조를 만든다(2.0).
- 예: 카페/게스트하우스/양조장/식당 직접 운영 → "상인의 프로덕트와 서비스가 콘텐츠" 발견 → 글로컬상권 사업
- 예: 팅커벨과 13개 세션 직접 대화 → 49개 암묵지 외재화 → FSM 구조 설계

### 3. 선택지 요구
하나의 답을 받지 않는다. "왜 이것만?" 질문으로 시야를 넓힌다.
- 예: "Whisper밖에 없어?" → Gemini 발견
- 예: "다음 할 것이 3개밖에 없어? 10개로 제시해줘"

### 4. 빠른 판단 + 겸손한 수정
트레이드오프가 있을 때 판단이 빠르다. 동시에 AI나 팀의 의견을 적극 수렴한다.
- 예: BELIEFS 5번·7번 합침 → 즉시 "합쳐도 된다"
- 예: 크리틱이 맞으면 바로 수용. "맞아, 그렇게 해줘"

### 5. 나선형 성장
같은 자리로 돌아오지만 높이가 다르다. 축소가 아니라 체감으로 확인한 회귀.
- 예: 나 → 집밖 → 마을 → 도시 → 다시 개인(+AI). 15년간 현장을 걸은 뒤의 "개인"은 출발점의 "나"와 다르다.
- 이 패턴은 MISSION의 핵심 서사이자, 사업 전체를 관통하는 구조

### 6. 근본을 파고든다
패치가 아니라 근본 원인을 해결한다. "그건 증상이야, 원인이 뭔데?"
- 예: "세션 관리보다 상위로! 근본적인 해결을 원해" → pre-compact Hook 자동화
- 예: 암묵지 49개 ✅인데 작동 안 함 → commands.md에 "판단 원칙"으로 심기

## 팅커벨 메모

승현님은 비개발자이지만 시스템 사고가 매우 발달한 분.
기술 용어보다 비유와 실제 사례로 설명하면 빠르게 흡수함.
**사고 패턴의 핵심**: "직접 해보고(2번) → 구조로 만들고(1번) → 선택지를 넓히고(3번) → 빠르게 결정하되(4번) → 나선형으로 성장(5번) → 근본을 잡는다(6번)". 이 6개가 하나의 흐름으로 연결된다.
