# 케이스 스터디 — 슈파 식자재 데이터분석 도구 v2

> 이승훈이 클로드 코드와 함께 식자재 공동구매 플랫폼 '슈파'의 데이터 분석 도구를 v1에서 v2로 전면 업그레이드한 이야기

---

## 시작 전 상태

- **코딩 경험**: 제로 (비개발자)
- **목표**: 식자재 공동구매 플랫폼 '슈파(SHUPA)'를 운영하면서, 파트너 데이터를 체계적으로 관리하고 공급업체 협상에 활용할 수 있는 도구를 만들고 싶었다
- **도구**: Windows 11 + 클로드 코드
- **v1 상태**: 기본적인 폼 입력 + 분석 테이블이 있었지만, 하나씩 입력해야 해서 불편하고 디자인이 딱딱했음

---

## 타임라인

### Day 1 — v2 전면 재작성 + 엑셀 가져오기 + 데이터 보호

**에피소드 1: "엑셀처럼 표에서 바로 입력하고 싶어요"**

v1의 가장 큰 불편함은 폼 기반 입력이었다. 품목 하나를 넣으려면 파트너명, 업종, 품목명, 수량, 단위, 단가를 각각 입력해야 했다. 파트너가 늘어날수록 이 작업이 반복되니까 비효율적이었다.

v2에서는 이렇게 바꿨다:
- 파트너를 탭으로 전환하고, 해당 파트너의 품목을 **엑셀 같은 표**에서 바로 입력
- Tab/Enter로 셀 이동, 마지막 행에서 자동 행 추가
- 파트너 추가/삭제가 탭 하나로 가능

**에피소드 2: "거래명세서를 엑셀로 받는데, 바로 넣을 수 없을까?"**

이미지 인식(OCR)으로 데이터를 넣는 건 어떻냐는 아이디어가 나왔다. 하지만 대화를 통해 실제로는 **엑셀 파일로 거래명세서를 받고 있다**는 걸 확인했다. 그래서 이미지 인식 대신 **엑셀 파일을 직접 읽어들이는 기능**을 만들었다.

- SheetJS 라이브러리로 브라우저에서 바로 엑셀 파일 파싱
- 열 매핑 UI: 어떤 열이 품목명/수량/단위/단가인지 선택 (헤더 자동 감지)
- 미리보기로 확인 후 한 번에 등록

이미지 인식보다 **100% 정확하고, 훨씬 빠르고, 오류가 없는** 방법이었다.

**에피소드 3: "다른 업체가 데이터를 보는 게 걱정돼요"**

비즈니스에서 핵심적인 질문이 나왔다: **파트너 간 데이터 보호**. 세 가지 걱정이 있었다.

1. 파트너 앞에서 화면을 보여줄 때 다른 파트너 데이터가 보이는 것
2. 복사/공유할 때 다른 업체 정보가 섞이는 것
3. 누군가 파일을 열어서 전체 데이터를 보는 것

세 가지를 모두 해결했다:
- **비밀번호 잠금**: 파일 열 때 비밀번호 필요, 자리 비울 때 즉시 잠금
- **발표 모드**: 선택한 파트너만 표시, 나머지 완전 숨김 (총구매량 보고서도 숨김)
- **복사 보호**: 파트너별 리포트 복사 시 해당 파트너만 포함

**에피소드 4: "파트너가 직접 입력하게 할 수 없을까?"**

이 질문은 미니 프로젝트의 한계를 넘는 순간이었다. 파트너별 로그인, 서버 데이터 저장, 권한 분리가 필요한 **풀 프로젝트** 영역이다. 지금은 v2를 실제로 써보면서 니즈를 구체화한 뒤, 나중에 풀 프로젝트로 전환하기로 했다.

## 최종 결과물

- **무엇을 만들었나**: 슈파 식자재 공동구매 데이터 분석 도구 v2
- **어떻게 작동하나**:
  - 파트너별 품목을 엑셀형 스프레드시트로 입력 (또는 엑셀 파일 가져오기)
  - 소비자(파트너)용 개인 주문 현황 리포트 → 클립보드 복사로 카톡 공유
  - 공급처용 전체 총구매량 정리 + CSS 바 차트 시각화
  - 협상 제안서 자동 생성 → 클립보드 복사/텍스트 파일 다운로드
  - 비밀번호 잠금 + 발표 모드로 데이터 보호
- **사용한 기술**: 단일 HTML 파일, Vanilla JS, CSS Grid/Flexbox, SheetJS(엑셀 파싱)

## 핵심 인사이트

### 1. 진짜 문제를 먼저 파악하자 — "이미지 인식" vs "엑셀 가져오기"

처음에는 "이미지 인식으로 데이터를 넣으면 어떨까?"라는 아이디어로 시작했다. 하지만 대화를 통해 실제로는 엑셀 파일로 데이터를 받고 있다는 걸 확인했다. 화려한 기술(OCR) 대신 정확하고 간단한 방법(엑셀 파싱)이 훨씬 효과적이었다. **사용자의 실제 워크플로우를 먼저 이해하는 게 가장 중요하다.**

### 2. 데이터 보호는 기술 문제가 아니라 비즈니스 문제

"다른 업체가 데이터를 보면 안 된다"는 요구사항은 코딩 지식에서 나온 게 아니라 **사업 운영 경험**에서 나왔다. 비개발자라도 비즈니스 맥락을 잘 전달하면, 클로드 코드가 적절한 기술적 해결책을 제안할 수 있다.

### 3. 미니 프로젝트의 한계를 아는 것도 성장이다

"파트너가 직접 입력하게 하고 싶다"는 요구사항이 나왔을 때, 이것이 미니 프로젝트로는 안 되는 영역이라는 걸 인식했다. 무리하게 구현하기보다 **현재 도구를 실전에서 써보면서 니즈를 구체화하고, 준비가 되면 풀 프로젝트로 전환**하는 단계적 접근이 더 효과적이다.

### 4. "만들면서 발견하는" 반복이 핵심

v1 → v2로 오면서 요구사항이 계속 구체화됐다. 처음부터 완벽한 기획서를 쓰는 게 아니라, 만들어보고 → 써보고 → 부족한 점을 발견하고 → 개선하는 사이클이 비개발자에게 가장 자연스러운 개발 방식이다.
