### 배운 개발 지식

- **하네스 (Harness)** — AI가 일할 환경을 코드 전에 먼저 설계하는 것. 말 마구처럼 AI가 제멋대로 뛰지 않게 방향을 잡아주는 틀. 4층 구조: CLAUDE.md(맥락/헌법) → spec.md(명세/계약서) → planning.md(계획/지도) → progress.md(기록/일지).
- **상위 CLAUDE.md (Upper Harness)** — 여러 MVP를 가진 브랜드/제품에서, 개별 MVP의 CLAUDE.md 위에 존재하는 브랜드 수준 컨텍스트 파일. "이게 슈파에 속하는가?"를 판단하는 기준이 됨.
- **피벗 병치 (Pivot Preservation)** — 전략이 여러 번 바뀌었을 때, 마지막 버전만 남기고 지우지 않는다. 네 버전을 모두 문서에 병치한다. 이유: 추측으로 통합하면 원래 맥락을 잃어버리고, 사용자만이 "지금 공식은 무엇인지" 결정할 수 있음.

- 📐 컨텍스트 엔지니어링 (Context Engineering) 🔄
  LLM은 기억이 없고, 컨텍스트 창이 제한되고, 확률적이라 일관성이 없다.
  선택지: ①매번 사람이 설명 반복 ②모델 성능 개선 기다리기 ③파일 시스템으로 외부 기억 만들기
  판단: ③. 메모리를 파일로, 판단 기준을 스펙으로, 진행 상황을 progress로. 모델 약점을 구조로 푼다.
  = LLM의 한계는 모델 업그레이드로 풀지 않고 파일 시스템으로 푼다. (단기 기억상실증 환자에게 포스트잇을 붙여주는 것과 같음)

- 📐 강제 로드 (Forced Context) ⚖️
  "이렇게 하세요"를 메모리에 적어두는 건 수동적 — AI가 읽어야만 작동한다.
  선택지: ①메모리에만 기록 (자주 잊힘) ②CLAUDE.md 같은 시스템 파일에 기록 (매 세션 강제 로드)
  판단: 시스템 파일. 이미 `C:\Users\leeha\CLAUDE.md` 같은 전역 CLAUDE.md가 세션 시작 프로토콜을 강제 로드하고 있음. 슈파-CLAUDE.md도 같은 원리.
  = 중요한 규칙은 "읽힐 가능성"이 아니라 "읽힐 수밖에 없는 위치"에 둔다. (냉장고 문에 붙인 쪽지와 서랍 속 쪽지의 차이)

- 📐 추측 금지 원칙 (No Hallucinated Context) 🔒
  대화에서 나온 것만 문서에 적고, 빈칸은 `[미결정]`으로 남긴다.
  선택지: ①있어 보이게 빈칸을 추측으로 채움 ②빈칸을 명시적으로 `[미결정]` 처리
  판단: ②. 추측으로 채운 내용은 다음 세션에서 사실로 읽히고 그 위에 또 추측이 쌓인다 → 몇 세션 만에 원래 방향과 전혀 다른 곳에 도착.
  = 모르는 건 "모른다"고 써두는 게 가짜 답보다 낫다. (수학 시험에서 모르는 문제를 찍는 것과 빈칸으로 두는 것 중, 감점이 아닌 내부 문서에선 빈칸이 정답)
