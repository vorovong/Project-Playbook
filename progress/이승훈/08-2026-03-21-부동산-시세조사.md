### 결과

- **급매물 수집기 API 전환** (`급매물_수집기_v2.py`, 90줄 추가/63줄 수정):
  - **엔드포인트 변경**: `new.land.naver.com/api/articles` → `m.land.naver.com/cluster/clusterList`
  - **수집 방식 변경**: 단일 목록 조회 → 2단계 구조
    1. 클러스터 조회 (지도 영역 내 매물 그룹)
    2. 클러스터별 개별 매물 상세 조회
  - **매물 유형 코드 변경**: PC 코드(`SG:SMS:DDDG`) → 모바일 코드(`D02:D03:E04`)
  - **Referer 변경**: `new.land.naver.com` → `m.land.naver.com`
  - **재시도 로직 강화**: 429 재시도 2→3회, 대기 30초→60초 점진 증가
  - **API 딜레이 증가**: 5초 → 8초

### 배운 개발 지식

- 📐 API 엔드포인트 다변화 (Endpoint Diversification) 🔄
  같은 서비스(네이버 부동산)도 PC/모바일/앱 버전마다 다른 API를 쓴다.
  선택지: ①차단된 PC API로 계속 시도 ②모바일 API로 전환 (다른 경로, 다른 제한 정책)
  판단: PC API는 완전 차단. 모바일 API는 클러스터 기반이라 요청 구조가 다르고, 429 임계치도 다를 수 있음.
  = 정문이 잠겨있으면 후문으로 가는 것. 같은 건물이지만 출입 규칙이 다르다.

- 📐 클러스터 기반 수집 (Cluster-based Crawling) 🗺️
  PC API는 "25구 × 페이지"로 순차 조회. 모바일 API는 "지도 영역 → 클러스터 → 매물" 2단계.
  클러스터 방식은 요청 수가 더 많지만, 한 요청당 데이터 양이 적어서 서버 부담이 낮고 차단 확률도 낮다.
  = 한 번에 큰 트럭으로 가는 것보다, 여러 번 오토바이로 나르는 게 규제에 안 걸린다.

### 어려웠던 것 / 다음에 할 것

- 클러스터 좌표(lat/lon/btm/lft/top/rgt) 계산이 필요해서 단순 시군구 코드만으로는 안 됨 — offset 0.06으로 영역 설정
- 모바일 API도 429가 걸릴 수 있으므로 딜레이+재시도 병행 필수
- 다음: 실제 수집 테스트 후 결과 검증, 딜레이 추가 강화 필요 여부 확인

  - OpenStreetMap 역지오코딩 (위도/경도 → 한글 주소)
  - 상가/사무실/건물 필터링, m²→평 자동 변환
  - 거래유형별 시트 분리 (매매/전세/월세)
  - 평단가 평균, 월세 평균 등 통계 자동 계산
  - 요약 시트 생성
- **부동산_시세조사.spec** — PyInstaller 설정 (exe 패키징용)

### 배운 개발 지식

- **API 크롤링 파이프라인** — 지도 API → 클러스터 조회 → 개별 매물 상세 조회라는 다단계 구조. 한 번에 모든 데이터를 가져올 수 없어서 단계별로 파고 들어가는 방식.
- **역지오코딩 (Reverse Geocoding)** — 위도/경도 좌표를 사람이 읽을 수 있는 주소로 바꾸는 것. OpenStreetMap의 무료 API 활용.
- **Tkinter GUI** — Python에 기본 내장된 화면 만들기 도구. 별도 설치 없이 바로 사용 가능.
- **PyInstaller** — Python 스크립트를 exe 파일로 만들어서 Python 없이도 실행 가능하게 하는 도구. 다른 사람에게 프로그램을 배포할 때 사용.

- 📐 데이터 수집 안정성 (Data Collection Stability) ⚖️
  네이버 부동산 API에서 데이터를 가져올 때 요청 속도 제한.
  선택지: ①빠르게 한번에(차단 위험) ②1초 딜레이(안정적이지만 느림)
  판단: 차단되면 아예 못 쓰니까 안정성 우선.
  = 급하게 달리다 넘어지면 더 느린 것. 천천히 확실하게.
